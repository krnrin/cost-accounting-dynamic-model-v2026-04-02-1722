<a id="组件归属混乱点6个"></a>### 2\.2 组件归属混乱点（6个）

<a id="混乱点1quotepage-同时承载报价和设变对比"></a>#### 🔴 混乱点1：QuotePage 同时承载”报价”和”设变对比”

__现状__：QuotePage\.tsx 导入 buildChangeComparisonTable、computeChangePricing、exportChangePricingExcel。

__问题__：页面标题是”报价”，但核心逻辑是”设变对比定价”。用户进入”报价”页面，看到的却是设变前后的成本差异表。与 ChangeEnginePage 功能重叠。

__归类方案__：QuotePage 专注于客户报价生成（输出报价单），设变对比归属 ChangeEnginePage。

<a id="混乱点2dashboardpage-承载过多功能"></a>#### 🔴 混乱点2：DashboardPage 承载过多功能

__现状__：DashboardPage 导入 KpiSection、ChartGrid、AllocationProgress、LifecyclePnLTable、HarnessProfitTable、AlertBanner、MetalImpactSummary。

__问题__：同时是”场景总览”和”成本核算结果展示”，与 QuotePage、GapAnalysisPage 功能边界不清。生命周期损益表（LifecyclePnLTable）更适合在”核价工作台”中展示。

__归类方案__：DashboardPage 专注”场景总览”（项目信息\+场景选择\+快捷入口），详细的成本核算结果移至”核价工作台”，分摊进度移至 AllocManagerPage。

<a id="混乱点3engineerworkbench-命名与实际功能不符"></a>#### 🟠 混乱点3：EngineerWorkbench 命名与实际功能不符

__现状__：导入 EngineSelector、HarnessCompareView、getRecentTraces。

__问题__：“工程师工作台”听起来是开发工具，实际功能是”成本追溯”和”引擎切换对比”。

__归类方案__：重命名为 “CostTracePage” 或 “EngineComparePage”。或将其功能拆分到 DashboardPage（成本追溯）和 SettingsPage（引擎选择）。

<a id="混乱点4定价功能分散在3个页面"></a>#### 🟠 混乱点4：定价功能分散在3个页面

__现状__：ConnectorPricingPage、WirePricingPage、DevPartPricingPage 结构相似但分散。

__归类方案__：新增 “PricingDashboardPage” 统一定价看板，3个页面作为定价看板的子Tab。

<a id="混乱点5包装功能分散在2个页面"></a>#### 🟠 混乱点5：包装功能分散在2个页面

__现状__：PackagingSchemePage 和 PackagingLogisticsPage 分开管理。

__归类方案__：合并为 “PackagingPage”，内部用Tab区分方案和费用。

<a id="混乱点6versionmanager-同时用于项目和场景"></a>#### 🟡 混乱点6：VersionManager 同时用于项目和场景

__现状__：/project/:id/versions（项目级）和 /project/:id/s/:sid/versions（场景级）共用同一组件。

__归类方案__：项目级版本管理 → 项目概览页内嵌。场景级版本管理 → DashboardPage内嵌或独立页面。

<a id="三种编码的关联关系"></a>#### 1\.3 三种编码的关联关系

连接器总成（功能列）          如: 2523328\-1（组合式充电插座）  
    │  
    ├── 散件1（零件号列）      如: 2377023\-2（端子TE）  
    │   └── SAP号             如: A02101947  
    │  
    ├── 散件2（零件号列）      如: 2377665\-4（防水栓TE）  
    │   └── SAP号             如: A03033091  
    │  
    └── 散件3（零件号列）      如: BH4309802（扎带）  
        └── SAP号             如: A03033097

__关键发现__: 一个连接器总成号（功能列）对应多个散件（零件号列），每个散件有独立的SAP号。三种编码形成”总成→散件→SAP”的树形关联。

// types/connectorLibrary\.ts  
  
/\*\* 连接器总成（功能层面） \*/  
interface ConnectorAssembly \{  
  id: string;  
  assemblyNo: string;        // 如 "2523328\-1"  
  name: string;              // 如 "组合式充电插座"  
  type: 'charging' | 'pdu' | 'motor' | 'battery' | 'other';  
  manufacturer: string;  
  voltage?: string;  
  current?: string;  
  pinCount?: number;  
  description?: string;  
  parts: AssemblyPart\[\];     // 关联的散件列表  
\}  
  
/\*\* 总成中的散件关联 \*/  
interface AssemblyPart \{  
  partId: string;  
  partNo: string;            // 如 "2377023\-2"  
  name: string;              // 如 "端子TE"  
  qty: number;  
  position?: string;  
  isCritical: boolean;  
  sapNo?: string;            // 如 "A02101947"  
\}  
  
/\*\* 连接器散件（零件层面） \*/  
interface ConnectorPart \{  
  id: string;  
  partNo: string;            // 如 "2377023\-2"  
  name: string;              // 如 "端子TE"  
  category: 'terminal' | 'seal' | 'housing' | 'accessory' | 'wire' | 'other';  
  manufacturer: string;  
  specification?: string;  
  unit: string;  
  unitPrice?: number;  
  sapNo?: string;            // 如 "A04030275"  
  description?: string;  
\}  
  
/\*\* 连接器信息库（前端使用） \*/  
interface ConnectorLibrary \{  
  assemblies: ConnectorAssembly\[\];   // 总成列表  
  parts: ConnectorPart\[\];            // 散件列表  
    
  // 索引方法  
  findAssemblyByNo\(assemblyNo: string\): ConnectorAssembly | undefined;  
  findPartByNo\(partNo: string\): ConnectorPart | undefined;  
  findPartBySapNo\(sapNo: string\): ConnectorPart | undefined;  
  findPartsByAssembly\(assemblyNo: string\): AssemblyPart\[\];  
\}

// 功能列输入时的智能补全  
function suggestAssemblyNo\(input: string\): Suggestion\[\] \{  
  // 搜索总成号 \+ 总成名称  
  const results = fuse\.search\(input, \{  
    keys: \['assemblyNo', 'name', 'manufacturer'\],  
    threshold: 0\.3,  
  \}\);  
    
  return results\.map\(r => \(\{  
    value: r\.item\.assemblyNo,           // 2523328\-1  
    label: \`$\{r\.item\.assemblyNo\} \- $\{r\.item\.name\}\`,  // "2523328\-1 \- 组合式充电插座"  
    detail: \`$\{r\.item\.manufacturer\} | $\{r\.item\.pinCount\}P | $\{r\.item\.voltage\}\`,  
    category: 'assembly',  
  \}\)\);  
\}  
  
// 用户选择总成号后的自动填充  
function onAssemblySelected\(assemblyNo: string\): AutoFillResult \{  
  const assembly = library\.findAssemblyByNo\(assemblyNo\);  
  return \{  
    // 自动填充零件名称  
    partName: assembly\.name,  
    // 自动填充关联散件列表（供零件号列选择）  
    relatedParts: assembly\.parts\.map\(p => \(\{  
      partNo: p\.partNo,  
      name: p\.name,  
      sapNo: p\.sapNo,  
      qty: p\.qty,  
    \}\)\),  
  \};  
\}

<a id="零件号列供应商散件料号"></a>##### 零件号列（供应商散件料号）

// 零件号列输入时的智能补全  
function suggestPartNo\(input: string, context?: \{ assemblyNo?: string \}\): Suggestion\[\] \{  
  // 如果有功能列已输入总成号，优先提示该总成的散件  
  if \(context?\.assemblyNo\) \{  
    const assembly = library\.findAssemblyByNo\(context\.assemblyNo\);  
    const parts = assembly?\.parts || \[\];  
      
    // 先搜索该总成的散件  
    const filtered = parts\.filter\(p =>   
      p\.partNo\.toLowerCase\(\)\.includes\(input\.toLowerCase\(\)\) ||  
      p\.name\.toLowerCase\(\)\.includes\(input\.toLowerCase\(\)\)  
    \);  
      
    if \(filtered\.length > 0\) \{  
      return filtered\.map\(p => \(\{  
        value: p\.partNo,                    // 2377023\-2  
        label: \`$\{p\.partNo\} \- $\{p\.name\}\`,   // "2377023\-2 \- 端子TE"  
        detail: \`SAP: $\{p\.sapNo\} | Qty: $\{p\.qty\}\`,  
        category: 'part\_in\_assembly',  
        sapNo: p\.sapNo,  
      \}\)\);  
    \}  
  \}  
    
  // 否则搜索全库散件  
  const results = fuse\.search\(input, \{  
    keys: \['partNo', 'name', 'manufacturer', 'sapNo'\],  
    threshold: 0\.3,  
  \}\);  
    
  return results\.map\(r => \(\{  
    value: r\.item\.partNo,  
    label: \`$\{r\.item\.partNo\} \- $\{r\.item\.name\}\`,  
    detail: \`$\{r\.item\.manufacturer\} | SAP: $\{r\.item\.sapNo\}\`,  
    category: 'part\_global',  
    sapNo: r\.item\.sapNo,  
  \}\)\);  
\}  
  
// 用户选择散件料号后的自动填充  
// 注：单价\(unitPrice\)由采购工程师维护，不在BOM编辑时填充  
// BOM中的单价从采购定价系统实时引用，开发工程师不可编辑  
function onPartSelected\(partNo: string\): AutoFillResult \{  
  const part = library\.findPartByNo\(partNo\);  
  return \{  
    partName: part\.name,           // 自动填充零件名称  
    supplier: part\.manufacturer,    // 自动填充供应商  
    unit: part\.unit,               // 自动填充单位  
    sapNo: part\.sapNo,             // 自动填充SAP号到回路号列  
    specification: part\.specification, // 自动填充规格  
    // unitPrice: 从采购定价系统引用，不自动填充  
  \};  
\}

<a id="回路号列sap号"></a>##### 回路号列（SAP号）

// 回路号列输入时的智能补全  
function suggestSapNo\(input: string\): Suggestion\[\] \{  
  // SAP号搜索（支持前缀匹配）  
  const results = fuse\.search\(input, \{  
    keys: \['sapNo', 'partNo', 'name'\],  
    threshold: 0\.2,  // SAP号精确度要求高  
  \}\);  
    
  return results\.map\(r => \(\{  
    value: r\.item\.sapNo,               // A04030275  
    label: \`$\{r\.item\.sapNo\} \- $\{r\.item\.name\}\`,  // "A04030275 \- 热缩管"  
    detail: \`料号: $\{r\.item\.partNo\} | 供应商: $\{r\.item\.manufacturer\}\`,  
    category: 'sap',  
    partNo: r\.item\.partNo,             // 关联的散件料号  
  \}\)\);  
\}  
  
// 用户选择SAP号后的自动填充  
// 注：单价由采购工程师维护，不在BOM编辑时填充  
function onSapSelected\(sapNo: string\): AutoFillResult \{  
  const part = library\.findPartBySapNo\(sapNo\);  
  return \{  
    partNo: part\.partNo,         // 自动填充零件号列  
    partName: part\.name,         // 自动填充零件名称  
    supplier: part\.manufacturer,  // 自动填充供应商  
    unit: part\.unit,  
    // unitPrice: 从采购定价系统引用，不自动填充  
  \};  
\}

    // 监听输入变化  
    const inputDisposable = univerAPI\.onEvent\('SheetEditChanged', \(params: any\) => \{  
      const \{ value \} = params;  
      if \(value && value\.length >= 2\) \{  
        const columnType = getColumnType\(params\.column\);  
        const results = searchConnectorLibrary\(value, columnType\);  
        setSuggestions\(results\);  
        setShowPanel\(results\.length > 0\);  
      \} else \{  
        setShowPanel\(false\);  
      \}  
    \}\);  
  
    // 监听编辑结束  
    const endDisposable = univerAPI\.onEvent\('SheetEditEnded', \(\) => \{  
      setShowPanel\(false\);  
    \}\);  
  
    return \(\) => \{  
      disposable?\.dispose\(\);  
      inputDisposable?\.dispose\(\);  
      endDisposable?\.dispose\(\);  
    \};  
  \}, \[univerAPI\]\);  
  
  return \{ suggestions, showPanel, editorPosition \};  
\};

<a id="链路1bom主表变更检测已实现"></a>#### 2\.1 链路1：BOM主表变更检测（已实现 ✅）

用户编辑BOM主表单元格  
    ↓  
UniverSheet onChange 回调  
    ↓  
handleSheetChange\(sheetData, sheetId\)  
    ↓  
【变更检测引擎】  
├── beforeRows = buildBomSheetRows\(变更前BOM\)  
├── afterRows  = buildBomSheetRows\(变更后BOM\)  
├── detection  = detectBomChanges\(beforeRows, afterRows\)  
│   └── 输出: \{ added\[\], removed\[\], modified\[\], hasChanges \}  
└── semantic   = classifyChangePatterns\(detection, hints\)  
    └── 输出: \{ pattern: 'replace'|'split'|'merge'|'wire\_spec\_replace'|\.\.\. \}

__变更检测粒度__： | 变更类型 | 检测方式 | 示例 | |———|———|——| | 新增行 | rowKey不匹配 | 新增零件PP3249999 | | 删除行 | rowKey不匹配 | 删除零件PP3241183 | | 修改字段 | 字段值对比 | 数量2→5、供应商A→B、单价10→12 |

__语义分类模式__： | 模式 | 判定条件 | 业务含义 | |——|———|———| | replace | 同组\+同供应商\+同单位\+行号距离≤3 | 物料替换（A→B） | | wire\_spec\_replace | 零件号前缀相同\+规格不同 | 导线规格变更（0\.5mm²→0\.75mm²） | | split | 1去\+N增\+同组 | 总成拆散件 | | merge | N去\+1增\+同组 | 散件合总成 | | fixed\_length | 单位M→PCS | 散裁定长化 | | modified | 数量/字段变化 | 普通字段修改 |

<a id="四关键缺陷分析"></a>### 四、关键缺陷分析

<a id="缺陷1联动是单向的bom主表派生表无反向联动"></a>#### 🔴 缺陷1：联动是单向的（BOM主表→派生表），无反向联动

__现状__： \- 用户在BOM主表编辑 → 派生表自动更新 ✅ \- 用户在assembly表编辑 → BOM主表不更新 ❌ \- 用户在secondary表编辑 → BOM主表不更新 ❌

__影响__： \- 如果用户在”总成散件清单”中修改了数量，BOM主表不会同步 \- 数据一致性风险：两个表的同一零件数量可能不一致

__复现步骤__： 1\. 打开BOM工作簿 2\. 在”总成散件清单”sheet中修改某行的数量 3\. 切换到对应的BOM主表sheet → 数量未更新

<a id="缺陷2每次变更需手动确认操作效率低"></a>#### 🔴 缺陷2：每次变更需手动确认，操作效率低

__现状__：

用户编辑BOM主表 → 检测变更 → 弹出CascadeConfirmWizard  
    ↓  
用户必须点击"确认" → 应用级联actions → 派生表更新

__问题__： \- 编辑1个单元格 → 弹出Wizard → 确认 → 才能继续编辑 \- 批量编辑时，每改一行都要确认一次，严重影响效率 \- 没有”自动应用”或”批量确认”模式

__用户场景__： > 工艺工程师需要一次性调整10个零件的工时，每次调整都要确认一次Wizard，操作繁琐。

<a id="缺陷3univer层面无原生公式联动"></a>#### 🟠 缺陷3：Univer层面无原生公式联动

__现状__： \- Sheet间联动通过__React状态管理__实现 \- Univer公式引擎__未启用__ \- 单元格之间没有公式引用关系

__对比__： | 方式 | 当前实现 | 理想实现 | |——|———|———| | 派生表数量 | React状态重建 | =VLOOKUP\(bomRow, 'BOM主表'\!A:G, 3, 0\) | | 成本重算 | 调用computeHarnessCost | Univer公式自动计算 | | 变更联动 | changeBus发布\-订阅 | Univer跨sheet公式引用 |

__影响__： \- 无法利用Univer的公式计算能力 \- 用户无法自定义公式 \- 跨sheet引用无法实时更新

<a id="缺陷4派生表全量重建大数据量性能差"></a>#### 🟠 缺陷4：派生表全量重建，大数据量性能差

__现状__：

<a id="优化2自动联动模式p0"></a>#### 5\.2 优化2：自动联动模式（P0）

__目标__：减少用户手动确认次数，提升操作效率

__方案__：

// 增加联动模式配置  
enum CascadeMode \{  
  MANUAL = 'manual',      // 每次变更都弹出Wizard（当前模式）  
  AUTO = 'auto',          // 自动应用级联，只记录变更履历  
  SEMI\_AUTO = 'semiAuto', // 自动应用简单变更，复杂变更弹出Wizard  
\}  
  
// 自动模式逻辑  
const shouldAutoApply = \(detection, semanticChanges\): boolean => \{  
  // 只有字段修改（无增删）→ 自动应用  
  if \(detection\.changes\.every\(c => c\.changeType === 'modified'\)\) return true;  
  // 只有1个新增/删除 → 自动应用  
  if \(detection\.changes\.length <= 1\) return true;  
  // 有replace/wire\_spec\_replace语义 → 自动应用  
  if \(semanticChanges\.some\(s => s\.pattern === 'replace' || s\.pattern === 'wire\_spec\_replace'\)\) return true;  
  // 其他情况（split/merge/多个增删）→ 弹出Wizard  
  return false;  
\};

__配置位置__： \- 在SettingsPage中增加”BOM联动模式”设置 \- 每个项目可独立配置联动模式

<a id="优化4增量更新p1"></a>#### 5\.4 优化4：增量更新（P1）

__目标__：避免全量重建派生表，提升性能

__方案__：

// 增量更新逻辑  
const applyIncrementalUpdate = \(prevRows, changes, semanticChanges\) => \{  
  const nextRows = \[\.\.\.prevRows\];  
    
  changes\.forEach\(change => \{  
    const rowIndex = nextRows\.findIndex\(r => r\.sourceBomRowKey === change\.rowKey\);  
      
    if \(change\.changeType === 'added'\) \{  
      // 只有新增行需要插入  
      nextRows\.splice\(rowIndex >= 0 ? rowIndex : nextRows\.length, 0, buildRowFromChange\(change\)\);  
    \} else if \(change\.changeType === 'removed' && rowIndex >= 0\) \{  
      // 只有删除行需要移除  
      nextRows\.splice\(rowIndex, 1\);  
    \} else if \(change\.changeType === 'modified' && rowIndex >= 0\) \{  
      // 只有修改行需要更新字段  
      nextRows\[rowIndex\] = \{ \.\.\.nextRows\[rowIndex\], \.\.\.getChangedFields\(change\) \};  
    \}  
  \}\);  
    
  return nextRows;  
\};

__实施要点__： 1\. 对比变更前后的差异，只更新变化的行 2\. 保持未变化行的引用稳定（减少React重渲染） 3\. 需要维护行索引映射

// authStore\.ts — 用户角色模型  
interface AuthUser \{  
  id: string;  
  email: string;  
  name: string;  
  role: string;  // ← 简单字符串，如 'ENGINEER'  
  // \.\.\.  
\}  
  
// usePermission\.ts — 权限检查Hook  
const \{ can, role \} = usePermission\(\);  
can\('simulation'\);  // ← 检查特定权限点  
  
// RoleGuard\.tsx — 组件级权限控制  
<RoleGuard field="bomEdit">          ← BOM编辑权限  
<RoleGuard field="deleteHarness">     ← 删除线束权限  
<RoleGuard field="costRates">         ← 成本费率权限

__当前问题__： \- 角色只有简单字符串（‘ENGINEER’）→ 无法区分部门权限 \- 飞书登录默认赋予 ‘ENGINEER’ → 所有飞书用户权限相同 \- 权限点未与业务因素关联 → 无法按16项成本因素控制 \- 缺少审批流权限 → 无法区分编辑/提交/审批/发布 \- 缺少数据范围权限 → 无法区分项目级/场景级/全局权限

<a id="审批流权限"></a>### 7\.9 审批流权限

【草稿】 ──保存草稿──→ 【已存档】 ──提交审批──→ 【审批中】 ──审批通过──→ 【已发布】  
    ↑                      │                        │  
    └──────────────────────┘                        └──审批驳回──→ 【草稿】  
  
状态流转权限：  
\- 草稿 → 已存档: 所有工程师  
\- 已存档 → 审批中: 部门工程师（只能提交自己负责的因素）  
\- 审批中 → 已发布: 部门负责人/PM  
\- 审批中 → 草稿: 部门负责人/PM（驳回）  
\- 已发布 → 草稿: 仅限PM/ADMIN（解锁，需理由）
