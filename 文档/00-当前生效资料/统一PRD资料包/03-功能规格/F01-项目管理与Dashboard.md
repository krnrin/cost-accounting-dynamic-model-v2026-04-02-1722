## 1. 功能定位

项目管理与 Dashboard 共同构成系统的一级入口层。

- 项目管理负责承载项目清单、项目识别、基础状态与进入动作。
- Dashboard 负责承载项目级经营总览、指标汇总与子模块导航。

这两个模块共同解决的问题不是“把页面放出来”，而是让用户能够从项目入口进入到一个稳定、可识别、可继续操作的经营工作面。

---

## 2. 业务目标

- 管理项目清单与项目基础状态
- 支撑项目搜索、筛选、新建、导入、删除等动作
- 在项目层形成统一经营入口
- 汇总关键经营指标并连接各专业模块
- 为场景、BOM、报价、分摊、跟踪等模块提供统一进入路径

---

## 3. 核心业务规则

### 3.1 项目是一级业务对象
所有场景、线束、BOM、报价、分摊、跟踪等能力都必须挂接在项目之下，而不是散落成平级对象。

### 3.2 项目列表负责入口，不替代专业模块
项目列表只负责识别、筛选、进入和基础状态呈现，不承载深度业务逻辑处理。

### 3.3 Dashboard 负责项目级汇总，不替代专业页面
Dashboard 是项目级总览和导航容器，不应替代 BOM、报价、设变、分摊、跟踪等专业页面。

### 3.4 项目状态应可支撑后续业务闭环
项目状态不应只是展示字段，而应能服务于后续场景推进、归档判断、经营状态识别。

---

## 4. 核心对象规格

## 4.1 项目对象
至少应包含：
- `projectId`
- `projectCode`
- `projectName`
- `customerName`
- `vehicleModel`
- `projectStatus`
- `harnessCount`
- `createdAt`
- `updatedAt`

## 4.2 Dashboard 汇总对象
建议独立抽象汇总数据结构，至少包含：
- `projectId`
- `summaryMetrics`
- `costSummary`
- `quoteSummary`
- `profitSummary`
- `warningSummary`
- `versionSummary`
- `updatedAt`

---

## 5. 页面与交互要求

### 5.1 项目列表页面
至少支持：
- 项目列表展示
- 搜索（项目名 / 编号 / 客户）
- 状态筛选
- 新建
- 导入
- 删除
- 进入项目详情

### 5.2 Dashboard 页面
至少支持：
- 项目关键指标展示
- 模块导航入口
- 当前项目经营摘要
- 场景、BOM、报价、Simulation、年降等主要入口

### 5.3 汇总信息要求
项目层至少应可看到：
- 项目状态
- 线束数量
- 成本与报价摘要
- 利润摘要
- 关键异常或预警摘要预留

---

## 6. 数据与计算要求

### 6.1 项目列表数据要求
项目列表应基于真实项目对象与动态汇总结果呈现，而不是静态演示数据。

### 6.2 Dashboard 数据要求
Dashboard 应优先读取统一业务对象的汇总结果，而不是在页面内重复拼凑各模块私有逻辑。

### 6.3 汇总边界
Dashboard 可汇总结果，但不负责取代各专业模块的明细与专业计算。

---

## 7. 模块边界

- 项目管理负责入口层，不负责深度业务计算。
- Dashboard 负责项目级汇总与导航，不负责明细编辑。
- 专业规则仍在 BOM、报价、分摊、设变、跟踪、预警等模块中承载。

---

## 8. 待确认项

1. 项目状态枚举是否继续细化
2. 项目层是否需要锁定 / 审批状态
3. Dashboard 是否首期承载版本摘要或预警摘要
4. Dashboard 是否支持场景切换总览

---

## 9. 开发实施建议

1. 保持项目对象稳定
2. 统一 Dashboard 汇总数据来源
3. 增强项目状态对后续闭环的支撑能力
4. 最后再深化管理摘要和异常摘要视图

---

## 10. 验收建议

### A 级验收
- 项目列表基础功能可用
- Dashboard 可进入、可展示关键指标
- 模块入口联通

### B 级验收
- Dashboard 汇总数据来源稳定
- 项目状态能支持后续模块判断
- 可承载预警 / 版本 / 场景摘要预留

### C 级验收
- 更重的管理视图整合
- 项目层审批或锁定机制

---

## 11. 结论

项目管理与 Dashboard 是整个平台的门面层与工作台入口层。它们的价值不在于页面本身，而在于让用户围绕“项目”这个统一对象进入系统，并在项目级视角下连接后续全部经营闭环能力。
