# 高压线束精算与决策引擎研发实施清单

> 版本：基于 `PRD_统一版_高压线束精算与决策引擎.md` 拆解的研发实施清单版  
> 目标：把统一 PRD 转换为可执行的研发实施视图，用于开发排期、任务拆解、联调与验收分级  
> 主依据：当前主应用 `D:/成本核算动态模型/app`、统一主 PRD、事实底稿  
> 使用原则：凡标记为“待确认”的内容，不直接视为当前版本已承诺交付范围

---

## 1. 使用说明

本清单不是替代统一 PRD，而是把统一 PRD 中的内容转成研发执行视图。研发使用时统一按四类判断：
- **已实现**：当前主应用已可直接确认，可作为复用、扩展、联调基础
- **待补闭环**：方向明确、已有页面/参数/历史实现线索，但当前仍缺完整数据链、状态流或规则闭环
- **待确认**：不可直接进入默认开发范围，需先产品拍板
- **建议顺序**：建议的研发推进先后，不代表必须一次性并行开发

---

## 2. 总体研发实施顺序

建议按以下顺序推进，而不是并行摊开：

### 第一阶段：稳住现有主链路
- 项目管理
- 项目详情 / Dashboard
- BOM 工作簿
- 客户报价工作台
- 系统设置
- Simulation / AnnualDrop 已有链路校验

**目标**：确认当前已实现能力的真实边界，形成稳定联调基线。

### 第二阶段：补执行闭环
- 场景管理主线
- 分摊管理与回收跟踪
- 设变管理
- 跟踪管理

**目标**：把“算完一次”升级为“持续跟踪执行状态”。

### 第三阶段：补治理与预警闭环
- 预警系统规则
- Alerts 页面化承载
- 版本 / 发布 / 审计 / 权限治理能力

**目标**：让系统具备可管理、可追溯、可提醒能力。

### 第四阶段：补产品化收口项
- Profile / 个人中心
- 工业蓝图风统一化改造
- 管理层视图深化
- 客户模板适配扩展

**目标**：完成产品层体验与治理收口，但不抢前面经营闭环主优先级。

## 4. 规则型研发任务清单

### 4.1 双引擎
- 明确内部实绩成本与客户报价的对象边界
- 明确二者共享哪些基础数据，哪些必须隔离快照
- 明确差异比较的统一口径

### 4.2 核算颗粒度
- 保证至少支持线束号 / BOM 行颗粒度
- 禁止回退为项目平均值或套级笼统核算

### 4.3 一次性费用分摊
- 保证按线束号独立、按根分摊
- 不得回退为平均打包分摊

### 4.4 分摊回收
- 保证装车比 × 累计产量链路可追踪
- 保证已回收 / 未回收状态可见

### 4.5 金属价格联动
- 保证铜 / 铝基准价参数化
- 明确联动公式入口与结果影响范围

### 4.6 进度价
- 明确其本质是协议价 vs 当前批量价差距追踪
- 不得实现成加权混合价

### 4.7 残余材料池
- 明确进入呆滞提报逻辑
- 不得继续计入当前产品成本

### 4.8 利润归因
- 保留 Shapley 归因方向
- 增补因果链瀑布图表达

## 7. 验收分级建议

### A 级：当前代码可直接验证
- 页面存在
- 路由已接入
- 基础功能可操作

### B 级：已有实现线索但需补闭环验证
- 有页面或参数入口
- 有历史巡检或历史测试证据
- 但当前仍需验证数据链、状态流、规则完整性

### C 级：仅可作为规划或待确认项管理
- 当前代码未确认
- 历史证据存在分歧
- 不进入当前版本已实现口径

---

## 8. 研发执行总原则

- 先对象，后页面。
- 先规则，后展示。
- 先闭环，后扩展。
- 先以当前 `app/` 代码为现实边界，再吸收历史快照作为补充证据。
- 所有“待确认”内容，在产品拍板前不直接当作已承诺需求开发。
- 所有“待补闭环”内容，优先补数据结构、状态流、规则来源和追溯链路。

如果后续要继续深化，建议下一步直接从本清单中再拆出四份专项功能规格：
1. 场景管理
2. 分摊回收跟踪
3. 设变 / 跟踪
4. 预警系统 / Alerts
