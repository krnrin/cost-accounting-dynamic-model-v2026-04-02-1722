## 1. 产品定义

### 1.1 产品名称
高压线束精算与决策引擎

### 1.2 产品定位
本产品面向汽车高压线束业务场景，定位为一套贯穿项目全生命周期的精算与经营决策平台。它不是单次报价输出工具，也不是只服务单一部门的静态核表工具，而是把成本核算、客户报价、设变管理、分摊回收、经营预警统一到同一业务事实体系中的核心系统。

### 1.3 一句话定位
在同一平台内实现成本算清、差异看见、报价做准、变更可控。

### 1.4 核心价值主张
产品的核心价值，不在于替代一张 Excel，而在于把过去分散在不同表格、不同角色、不同阶段中的关键经营动作收拢到统一平台中，形成一套可持续运行的经营机制。它集中承载：
- 内部实绩成本穿透
- 客户报价决策与输出
- 设变与年降管理
- 一次性费用分摊与回收跟踪
- 预警、比较、回溯与经营判断

统一后的主 PRD 也不只是需求汇总文档，而是同时承担三种职责：
- 对管理层说明产品为什么存在、解决什么经营问题
- 对产品与业务说明规则边界、状态边界与协作边界
- 对开发说明哪些能力已被程序确认、哪些能力仍处于待收口阶段

### 1.5 产品边界
- `cloud-code` 仓库不是成本核算系统本体。
- 成本核算系统主项目以 `D:/成本核算动态模型` 为准。
- 程序完成状态以当前主应用 `D:/成本核算动态模型/app` 的真实代码为主依据。
- 历史会话、历史截图、历史验证结果用于补充判断产品演进与实现线索，但不能替代当前主应用代码现实。

## 2. 业务背景与问题定义

### 2.1 行业背景
高压线束业务天然具有高复杂度和高敏感度：一方面，BOM 层级深、物料类型多、连接器和导线价值集中，决定了成本分析不能停留在项目平均值；另一方面，铜、铝等金属价格波动、客户设变频繁、项目生命周期长，又使得报价和经营判断始终处于动态变化中。

与此同时，一次性费用如工装、模具、试验、研发等，往往具有投入早、回收慢、影响长尾的特征。如果仍然采用静态表格、笼统分摊或单次核价的方式，系统就很难真实反映项目不同阶段的经营状态。

### 2.2 当前业务问题
当前业务面临的核心问题，不是“算不出结果”，而是“算出的结果难以支撑持续经营判断”。主要表现为：
- 静态 Excel 或单表工具难以承载多版本、多场景与持续回溯
- 按套笼统分摊会掩盖单线束真实盈利情况
- 客户报价与内部实绩成本割裂，难以同时看清真实成本与客户口径
- 设变、年降、协议价落实、一次性费用回收等关键过程缺乏连续跟踪
- 金属价格、费率、BOM 变化后，缺少统一的比较、预警与回溯机制
- 财务、销售、工艺、管理层之间缺乏统一的数据依据与状态依据

### 2.3 建设目标
系统建设的目标，不是再替代一张表，而是把高压线束项目从“阶段性核价动作”升级为“持续经营管理机制”，使业务能够：
- 在单线束号 / BOM 行颗粒度看清成本与利润
- 在内部实绩成本与客户报价之间建立可比较、可解释的双引擎体系
- 在项目不同阶段之间保留场景化事实与演进记录
- 在执行过程中持续跟踪分摊回收、设变影响、年降影响与金属联动影响
- 在关键异常和经营拐点出现时及时预警并支撑决策

## 3. 用户角色与协作机制

### 3.1 核心角色
系统服务的并不是单一岗位，而是一组围绕项目经营共同工作的角色：

#### 财务
- 负责费率基准、利润要求和成本结构审核
- 关注内部实绩成本、分摊回收、利润穿透与版本发布
- 在历史规则中具备较强的基准发布与审核属性【已确定未完全实现】

#### 销售
- 负责客户报价口径、客户模板输出和商务参数维护
- 关注客户报价结果、调价空间、协议价落实和客户变更影响
- 可调参数边界与财务锁定边界仍需进一步明确【待确认】

#### 工艺 / IE / 制造相关角色
- 提供工时、制造条件、工厂效率、包装物流等业务输入
- 影响内部实绩成本核算与项目执行假设

#### 管理层
- 关注项目利润、预警、对比、回收进度、版本演进和异常闭环
- 更关注趋势、差异与经营结果，而非逐行填表过程

#### 产品 / 开发
- 负责把规则沉淀为系统能力
- 保证文档规则、历史决策与程序实现的一致性

### 3.2 协作原则
系统的协作价值，不在于增加更多沟通动作，而在于让财务、销售、工艺和管理层围绕同一套业务事实工作。统一核算口径、统一状态跟踪、统一版本依据，是系统协作成立的前提。

### 3.3 权限与治理方向
从历史材料看，系统已经不仅仅追求“算得出来”，而是在逐步走向“算得清、管得住、追得回”。目前较稳定的治理方向包括【已确定未完全实现】：
- 财务掌握基准费率发布权
- 销售维护商务报价映射，但不应随意修改实绩参数
- 流程 / 管理角色具有审批、合规或审计视角
- 关键版本应具备唯一追踪标识与流程状态

具体权限边界仍需产品化确认【待确认】。

## 4. 产品整体方案

### 4.1 方案框架
产品整体遵循“双引擎 + 多场景 + 决策支持”的方案框架：
- 双引擎：同时管理内部实绩成本与客户报价结果
- 多场景：在同一项目下承载不同业务阶段的独立经营事实
- 决策支持：让系统具备比较、追踪、预警、回溯能力，而不是只完成一次性输出

### 4.2 双引擎体系【已确定未完全实现】
双引擎是产品区别于传统报价工具的核心结构。

#### 内部实绩成本引擎
用于还原企业内部真实成本口径，包括材料、人工、制造、管理、一次性费用分摊、包装物流等。

#### 客户报价引擎
用于形成面向客户的报价结果，包括客户口径的参数、模板、利润要求和可调策略。

#### 双引擎价值
同一条线束需要同时看到：
- 内部真实成本
- 客户报价结果
- 二者差异与利润空间

### 4.3 多场景体系【已确定未完全实现】
多场景不是附加功能，而是项目经营管理的主线承载方式。统一 PRD 采用以下场景主线：
- 初始报价
- 定点
- 设变
- 年降

每个场景应独立持有：
- 生命周期假设
- 产量
- 费率
- BOM
- 分摊状态
- 报价参数
- 版本状态

场景之间的数据继承机制尚未完全确认【待确认】。

### 4.4 决策支持体系
产品价值最终要落在经营判断上，因此系统必须具备：
- 比较：不同场景、不同版本、不同参数之间比较
- 追踪：分摊回收、协议价落实、设变影响、年降执行
- 预警：金属价格、分摊回收、异常超支等关键节点提醒
- 回溯：项目、场景、版本、关键参数变化的追溯

## 5. 核心业务对象与数据模型

### 5.1 核心业务对象
本产品的数据模型，不以“表单字段”组织，而以真实业务对象组织。核心对象包括：

#### 项目【已实现】
项目是管理的顶层对象，承载客户、车型、项目状态、线束集合等基础信息。

#### 场景【已确定未完全实现】
场景是承载生命周期差异的核心对象。统一 PRD 中把场景视为与项目绑定、但彼此独立的业务容器。

#### 线束号【已确定未完全实现】
线束号是成本与报价核算的核心分析对象，系统应以单线束号而非项目整体平均值进行分析。

#### BOM 行【已确定未完全实现】
BOM 行是更细颗粒度的成本承载对象，逐行物料和逐层汇总是关键规则之一。

#### 费用项【已确定未完全实现】
费用项包括但不限于：
- 材料成本
- 直接人工
- 制造费用
- 管理费用
- 包装物流
- 一次性费用
- 利润要求

#### 费率基准【已确定未完全实现】
用于驱动人工、制造、管理、利润等关键系数和口径。

#### 版本【已确定未完全实现】
版本用于承载费率、BOM、财务快照、资本验证、发布时间线等状态事实。

### 5.2 核心数据关系
统一 PRD 约束的不是孤立字段，而是对象之间的关系：
- 一个项目可包含多个场景【已确定未完全实现】
- 一个场景可包含多个线束号
- 一个线束号包含多条 BOM 行
- 每个场景关联独立的费率、产量、装车比、分摊状态和报价参数
- 每个版本应具备可回溯属性【已确定未完全实现】

### 5.3 多项目复用方向
历史材料已显示系统从 G281 单项目逐步走向 E281/G281 等多项目复用：
- 项目切换器
- 独立数据域 / 独立 IndexedDB
- 项目注册表
- 锁定与变更流程

这说明统一 PRD 在设计上应支持项目级隔离与项目复用，而不是被单项目演示样本所绑定【已确定未完全实现】。

### 5.4 面向开发的对象落地约束
为避免对象定义停留在概念层，统一 PRD 对开发至少提出以下落地约束：

#### 项目对象
至少应承载：
- 项目标识
- 客户 / 车型 / 项目阶段
- 线束集合
- 项目状态
- 创建与更新时间

#### 场景对象
至少应承载：
- 所属项目
- 场景类型（初始报价 / 定点 / 设变 / 年降）
- 生命周期假设
- 产量与装车比
- 费率快照
- BOM 版本引用
- 报价参数快照
- 场景状态

#### 线束号对象
至少应承载：
- 线束号标识
- 所属场景
- 产量 / 装车关系
- 关联 BOM 行集合
- 成本结果
- 报价结果

#### BOM 行对象
至少应承载：
- 料号 / 名称 / 类别
- 用量
- 单价或价格来源
- 金属属性或材料属性
- 替代 / 取消 / 新增等变更状态
- 对应线束号

#### 版本对象
至少应承载：
- 版本标识
- 关联对象范围（项目 / 场景 / BOM / 费率）
- 生成时间
- 生成人
- 版本状态
- 可追溯备注或差异说明

这些约束的目的，不是一次性穷举所有字段，而是保证后续设计、建模、页面和计算逻辑都围绕同一批稳定业务对象展开，避免回到“页面先行、字段后补”的实现方式。

- 主题偏好
- 会话 / 使用相关配置

具体职责边界仍待确认【待确认】。

### 6.15 面向开发的模块交付边界
为提升研发可落地性，统一 PRD 对模块边界作如下补充约束：
- 项目管理负责项目入口、列表与基础状态，不承担深度核算逻辑。
- Dashboard 负责项目级汇总与导航，不替代 BOM、报价、跟踪等专业页面。
- BOM 工作簿负责明细数据承载与编辑，不直接承担管理决策展示职责。
- 客户报价工作台负责客户口径输出与差异比较，不替代内部实绩成本口径本身。
- Simulation / AnnualDrop 负责情景测算与周期影响分析，不应与正式版本发布流程混写。
- Alloc / Change / Tracking 属于执行闭环模块，应以业务事件跟踪、状态推进、影响记录为核心。
- 预警系统负责规则与触发机制，Alerts 负责页面承载与查看入口，两者不能混为一个概念。
- Settings 负责参数配置与口径管理，不应演变为脱离业务对象的泛化配置堆场。

开发实现时，新增页面、状态结构、计算逻辑与接口设计都应优先回答一个问题：该能力究竟属于哪个业务对象、哪个模块边界、哪个经营阶段。如果回答不清，就说明设计仍未收口。

## 7. 关键业务规则与计算口径

### 7.1 双引擎规则【已确定未完全实现】
系统采用“内部实绩成本 + 客户报价”的双线核算口径。

要求：
- 同一条线束同时看到内部真实成本与客户报价结果
- 允许从利润穿透与调价判断两个角度分析差异

### 7.2 核算颗粒度规则【已确定未完全实现】
核算颗粒度以单线束号 / BOM 行为主，而不是项目平均值或套级笼统核算。

要求：
- 逐条线束分析
- 逐行 BOM 分析
- 逐层汇总结果

### 7.3 一次性费用分摊规则【已确定未完全实现】
一次性费用（工装、试验、研发等）必须按线束号独立、按根分摊。

这是相对旧版“按套平均分摊”的关键升级点。

### 7.4 分摊回收跟踪规则【已确定未完全实现】
系统需要基于装车比 × 累计产量跟踪分摊回收进度。

要求：
- 能看到已回收 / 未回收状态
- 分摊完毕后可提供调价提醒

### 7.5 多场景规则【已确定未完全实现】
统一 PRD 明确采用一个项目多场景模型：
- 初始报价
- 定点
- 设变
- 年降

要求：
- 各场景独立持有关键业务事实
- 支持场景比较和演进追踪

场景继承机制仍待确认【待确认】。

### 7.6 金属价格联动规则【已确定未完全实现】
铜 / 铝价格是成本敏感因素，系统必须支持金属价格联动。

当前程序中已看到金属基准价设置入口，因此至少已有参数入口，但完整规则闭环仍需继续核准。

### 7.7 进度价规则【已确定未完全实现】
历史补丁指南已明确：
- 进度价不应理解为加权混合价
- 其本质是协议价与当前批量价的差距追踪

该规则对跟踪模块设计有直接影响。

### 7.8 残余材料池规则【已确定未完全实现】
历史补丁指南已明确：
- 未匹配 / 取消料号的残余材料池，本质应进入呆滞提报流程
- 不应继续分摊到当前产品成本中

### 7.9 利润归因规则【已确定未完全实现】
历史补丁指南显示：
- 系统需要同时兼顾数学归因与业务可解释性
- 应保留 Shapley 归因，同时增加因果链瀑布图

### 7.10 关键参数配置化规则【已确定未完全实现】
金属敏感度、材料组成、状态默认值等关键参数应逐步配置化，而非长期硬编码。

### 7.11 权限与发布治理规则【已确定未完全实现】
历史验收清单已出现：
- 基准发布流
- Audit Trace ID
- 异常关闭机制
- 权限隔离

统一 PRD 将这些视为正式产品治理方向，但具体系统化实现仍需继续核准。

### 7.12 面向开发的规则落地要求
为避免业务规则在研发过程中被弱化为页面文案或静态展示，统一 PRD 补充以下落地要求：
- 规则应优先沉淀在可复用的计算、校验、状态流或配置结构中，而不是散落在页面条件分支中。
- 同一规则若同时影响 BOM、报价、跟踪、预警，应以统一来源管理，避免多处复制后口径漂移。
- 所有“已确定未完全实现”的规则，在开发拆解时都应明确：当前已落地部分、尚缺闭环部分、验证方式。
- 所有“待确认”规则，在未完成产品确认前，不应被写死为默认逻辑，更不应在多个模块内各自实现不同版本。
- 所有涉及版本、审批、发布、追溯的能力，应优先保证状态一致性，再补充页面体验层表达。

开发视角下，规则是否真正落地，不以是否有页面入口为准，而以是否形成了稳定的数据结构、计算过程、状态流和验证依据为准。

## 8. 交互与体验原则

### 8.1 体验目标
系统的交互目标不是“功能堆叠”，而是让业务在最短路径内看清：
- 核心成本
- 利润差异
- 回收进度
- 预警状态
- 版本变化

### 8.2 设计风格方向【已确定未完全实现】
历史记忆与部分会话中，产品体验已明确朝“工业蓝图风”推进：
- 去彩色渐变
- 去黑色图标方块
- 弱化背景干扰
- 强调信息层级、数据密度与工业治理感

但当前主应用 `app/src/index.css` 仍可见较强的 glassmorphism / green-future 风格，因此该方向目前更适合表述为“已确定未完全实现”。

### 8.3 信息优先原则
在体验层面，系统优先服务判断效率，而不是视觉炫技：
- 数据优先于装饰
- 层级优先于炫技
- 决策信息优先于视觉噱头
- 关键异常应优先暴露

### 8.4 页面组织原则
历史材料已显示四页心智模型：
- 预演
- 核算
- 跟踪
- 归档

即使当前主应用实现方式不同，这种组织方式仍然说明，产品不是围绕单页面展开，而是围绕业务闭环组织信息【已确定未完全实现】。

## 9. 实施边界与迭代路线

### 9.1 当前版本主线
当前阶段的产品主线，应聚焦在“先把核心经营闭环跑通”，重点围绕以下能力展开：
- 项目管理与项目详情
- BOM 与报价工作台
- 模拟 / 年降
- 设置与基础参数管理
- 分摊、跟踪、设变等执行型模块的持续完善
- 预警系统与个人中心的产品化收口

### 9.2 程序完成状态的分层表达
为了避免文档与现实脱节，新 PRD 必须明确区分三层：

#### 主程序现实
以当前 `app/` 代码为准，能确认的页面、入口、路由、配置、数据结构与功能。

#### 历史实现线索
来自文档、历史会话、Playwright 巡检、历史测试和历史截图等证据。

#### 目标能力 / 规划能力
已进入产品范围，但当前未在主程序中确认完整实现的能力。

### 9.3 近期重点【已确定未完全实现】
- 场景管理闭环
- 分摊回收跟踪闭环
- 预警系统规则与展示闭环
- 设变 / 年降的版本管理深化
- 个人中心边界收口

这些重点的共同目标，不是继续堆模块数量，而是把“已进入产品范围但尚未完全闭环”的能力，逐步收敛为可执行、可验证、可汇报的稳定能力。

### 9.4 非目标项
以下内容不应成为当前阶段产品主线：
- 仅为展示而展示的炫技型页面
- 脱离真实业务规则的泛化配置平台
- 脱离项目 / 场景 / 线束 / BOM 事实的轻量填表工具

### 9.5 面向开发的实施拆解原则
后续研发拆解建议统一遵循以下顺序：
1. 先确认业务对象与规则归属，再设计页面与交互。
2. 先确认当前主程序已实现边界，再决定是补闭环、补页面还是补规则。
3. 先实现能形成经营闭环的主链路，再补充展示层、汇报层与偏体验层能力。
4. 对“已确定未完全实现”的模块，优先补齐数据结构、状态流、校验和追溯链路。
5. 对“待确认”的模块或规则，先保留扩展位与文档标记，不直接假设为已拍板需求。

这意味着统一 PRD 在研发执行时，更适合作为“范围控制与边界判断依据”，而不是被拆成一组互相独立、缺乏统一口径的页面需求。

## 10. 待确认项

以下事项在统一 PRD 中明确保留为待确认，不直接写成正式落地结论。其原则是：凡证据不足、版本线索不一致、或当前主程序尚未核准的内容，一律先降级处理。

1. **场景之间的数据继承机制**
   - 新建场景时继承哪些字段
   - 哪些字段允许复制，哪些字段必须重新确认

2. **报价引擎可调参数的权限边界**
   - 哪些由销售可调
   - 哪些由财务锁定
   - 哪些需审批后调整

3. **预警系统分级标准**
   - 阈值
   - 颜色
   - 触发逻辑
   - 展示优先级

4. **个人中心 / Profile 职责边界**
   - 仅承载个人偏好
   - 还是承载权限、通知、审计入口等更重功能

5. **设变与年降的版本管理深度**
   - 是否要求版本级差异对比
   - 是否要求审批状态或冻结状态

6. **客户模板适配范围与优先级**
   - 首期覆盖哪些模板
   - 模板与系统字段如何映射

7. **Alerts / Profile 在当前主应用中的实际落地深度**
   - 历史会话已有推进证据
   - 但当前主应用代码尚未形成稳定确认结论

### 10.1 待确认项的处理原则
为避免待确认项在研发过程中被误当成既定需求，统一 PRD 进一步约束：
- 待确认项可以进入设计预留，但不应直接进入默认业务逻辑。
- 待确认项若影响多个模块，必须先统一口径，再进入开发实施。
- 待确认项若已经存在历史实现线索，应保留证据说明，但不能据此提升为“已实现”。
- 待确认项转为正式需求后，应同步更新状态标记、影响模块与验证依据。

## 12. 结论
统一后的产品定义应明确：

> 高压线束精算与决策引擎，不再是一次性报价输出工具，而是一套面向高压线束项目全生命周期的场景化成本与经营决策平台。

它要解决的，不是“再做一张更复杂的表”，而是把过去分散在报价、核算、设变、分摊、跟踪和经营判断中的关键动作，统一到同一套业务事实与管理机制中。其核心价值在于：
- 把内部实绩成本与客户报价统一到同一业务事实中
- 把单线束号 / BOM 行颗粒度作为核心分析基础
- 把分摊回收、设变、年降、金属联动、预警、回溯纳入持续运行的管理闭环
- 把程序现状、历史决策与产品目标统一到一份可开发、可汇报、可持续迭代的主 PRD 中

统一 PRD 的价值，也不只在于当前这一版文档本身，而在于它为后续版本迭代建立了统一口径：哪些是已确认的产品事实，哪些是已确定但尚未完全收口的能力，哪些仍需继续核准。只有保持这种分层表达，产品、开发、业务与管理层才能围绕同一现实持续推进，而不是在不同版本认知之间反复切换。
