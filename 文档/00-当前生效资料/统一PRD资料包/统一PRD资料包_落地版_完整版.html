源文件：02-研发拆解/PRD_统一版_研发实施清单.md

高压线束精算与决策引擎研发实施清单

版本：基于 PRD_统一版_高压线束精算与决策引擎.md 拆解的研发实施清单版
目标：把统一 PRD 转换为可执行的研发实施视图，用于开发排期、任务拆解、联调与验收分级
主依据：当前主应用 D:/成本核算动态模型/app、统一主 PRD、事实底稿
使用原则：凡标记为“待确认”的内容，不直接视为当前版本已承诺交付范围

1. 使用说明

本清单不是替代统一 PRD，而是把统一 PRD 中的内容转成研发执行视图。研发使用时统一按四类判断：

  • 已实现：当前主应用已可直接确认，可作为复用、扩展、联调基础
  • 待补闭环：方向明确、已有页面/参数/历史实现线索，但当前仍缺完整数据链、状态流或规则闭环
  • 待确认：不可直接进入默认开发范围，需先产品拍板
  • 建议顺序：建议的研发推进先后，不代表必须一次性并行开发

2. 总体研发实施顺序

建议按以下顺序推进，而不是并行摊开：

第一阶段：稳住现有主链路

  • 项目管理
  • 项目详情 / Dashboard
  • BOM 工作簿
  • 客户报价工作台
  • 系统设置
  • Simulation / AnnualDrop 已有链路校验

目标：确认当前已实现能力的真实边界，形成稳定联调基线。

第二阶段：补执行闭环

  • 场景管理主线
  • 分摊管理与回收跟踪
  • 设变管理
  • 跟踪管理

目标：把“算完一次”升级为“持续跟踪执行状态”。

第三阶段：补治理与预警闭环

  • 预警系统规则
  • Alerts 页面化承载
  • 版本 / 发布 / 审计 / 权限治理能力

目标：让系统具备可管理、可追溯、可提醒能力。

第四阶段：补产品化收口项

  • Profile / 个人中心
  • 工业蓝图风统一化改造
  • 管理层视图深化
  • 客户模板适配扩展

目标：完成产品层体验与治理收口，但不抢前面经营闭环主优先级。

3. 分模块研发实施清单

3.1 项目管理

状态：已实现

当前依据：项目列表、搜索、状态筛选、新建、导入、删除已可确认

当前可直接复用

  • 项目列表入口
  • 搜索与状态筛选
  • 新建 / 导入 / 删除基础动作

后续研发重点

  • 校验项目对象字段是否已满足多项目复用要求
  • 校验项目状态是否能支撑场景、跟踪、归档等后续链路
  • 为后续场景管理预留稳定挂载关系

待确认

  • 项目状态枚举是否还要继续细化
  • 项目层是否需要审批或锁定状态

建议顺序

高，作为所有后续模块挂载基础

3.2 项目详情 / Dashboard

状态：已实现

当前依据：DashboardPage 已存在并接入主链路

当前可直接复用

  • 项目级经营视图入口
  • 子模块导航容器
  • 基础指标汇总展示

后续研发重点

  • 明确 Dashboard 只做汇总与导航，不承接深度规则编辑
  • 明确哪些指标来自内部实绩成本、哪些来自客户报价
  • 为场景切换、版本查看、预警摘要预留统一入口

待确认

  • Dashboard 是否承担版本对比总览
  • Dashboard 是否承担预警聚合视图

建议顺序

高，作为后续模块整合枢纽

3.3 BOM 工作簿 / BOM 管理

状态：已实现（页面与主入口层面）

当前可直接复用

  • BOM 页面骨架
  • 明细查看与编辑入口
  • 逐行数据承载能力

后续研发重点

  • 校验 BOM 行对象是否已具备材料属性、金属属性、变更状态等关键字段
  • 打通 BOM 行与线束号、场景、版本的稳定关系
  • 明确 BOM 变更如何向报价、分摊、跟踪模块传导

待补闭环

  • BOM 变更影响链路
  • 版本差异比较
  • 变更状态对残余材料池、设变管理的影响映射

待确认

  • BOM 分类规则与客户模板字段映射深度

建议顺序

高，属于核算主数据基础

3.4 客户报价工作台

状态：已实现（页面与主入口层面）

当前可直接复用

  • 报价页面入口
  • 客户口径结果承载页面
  • 与内部结果比较的基础位置

后续研发重点

  • 明确客户报价参数快照与内部实绩参数快照的分离方式
  • 梳理销售可调、财务锁定、需审批三类参数边界
  • 明确报价输出与客户模板映射关系

待补闭环

  • 客户口径参数治理
  • 报价结果与版本/场景的绑定关系
  • 调价空间与利润差异解释

待确认

  • 客户模板首期覆盖范围
  • 报价引擎权限边界

建议顺序

高，属于双引擎核心之一

3.5 Simulation / 决策仿真

状态：已实现（页面与入口层面）

当前可直接复用

  • 参数调整与结果观察入口
  • 模拟视图页面骨架

后续研发重点

  • 明确 Simulation 属于假设测算，不等于正式版本发布
  • 明确其与场景、版本、年降、金属联动之间的关系
  • 控制模拟结果与正式经营事实之间的边界

待补闭环

  • 模拟结果保存/引用规则
  • 模拟参数来源与快照机制

待确认

  • 模拟结果是否允许转正式场景或正式版本

建议顺序

中，依赖前面对象与规则清晰化

3.6 年降管理

状态：已实现（页面与入口层面）

当前可直接复用

  • 年降页面入口
  • 周期影响观察容器

  • 明确年降属于独立场景、独立版本还是附着在既有场景中的状态
  • 打通年降对成本、报价、利润、跟踪的影响链路

待补闭环

  • 年降版本管理
  • 年降规则与跟踪结果联动

待确认

  • 年降版本管理深度

4. 规则型研发任务清单

4.1 双引擎

  • 明确内部实绩成本与客户报价的对象边界
  • 明确二者共享哪些基础数据，哪些必须隔离快照
  • 明确差异比较的统一口径

4.2 核算颗粒度

  • 保证至少支持线束号 / BOM 行颗粒度
  • 禁止回退为项目平均值或套级笼统核算

4.3 一次性费用分摊

  • 保证按线束号独立、按根分摊
  • 不得回退为平均打包分摊

4.4 分摊回收

  • 保证装车比 × 累计产量链路可追踪
  • 保证已回收 / 未回收状态可见

4.5 金属价格联动

  • 保证铜 / 铝基准价参数化
  • 明确联动公式入口与结果影响范围

4.6 进度价

  • 明确其本质是协议价 vs 当前批量价差距追踪
  • 不得实现成加权混合价

4.7 残余材料池

  • 明确进入呆滞提报逻辑
  • 不得继续计入当前产品成本

4.8 利润归因

  • 保留 Shapley 归因方向
  • 增补因果链瀑布图表达

5. 治理型研发任务清单

当前已明确方向，但多为待补闭环

  • 费率基准发布流
  • Audit Trace ID
  • 权限隔离
  • 异常关闭机制
  • 版本追溯
  • 多工厂对标

研发建议

  • 不要把治理能力放到最后才考虑
  • 至少先定义数据结构、状态枚举和追溯字段
  • 首期即使不做完整审批 UI，也应先留出可扩展的版本状态与审计字段

6. 待确认项转研发动作映射

可先设计预留、不能直接开发写死

  • 场景继承机制
  • 报价引擎权限边界

  • 先做单点产品确认，再进入模块开发

7. 验收分级建议

A 级：当前代码可直接验证

  • 页面存在
  • 路由已接入
  • 基础功能可操作

B 级：已有实现线索但需补闭环验证

  • 有页面或参数入口
  • 有历史巡检或历史测试证据
  • 但当前仍需验证数据链、状态流、规则完整性

C 级：仅可作为规划或待确认项管理

  • 当前代码未确认
  • 历史证据存在分歧
  • 不进入当前版本已实现口径

功能规格-项目管理与Dashboard

适用范围：高压线束精算与决策引擎
来源依据：PRD_统一版_高压线束精算与决策引擎.md、PRD_统一版_研发实施清单.md
文档目标：把“项目管理 + Dashboard”从 PRD 级表述下钻为可供研发拆解的专项功能规格
当前判断：属于当前主应用已实现的核心入口能力

交叉引用

  • 对应总纲章节：1. 项目管理与Dashboard
  • 对应开发任务书：04-开发任务书/T01-项目管理与Dashboard.md

1. 功能定位

项目管理与 Dashboard 共同构成系统的一级入口层。

  • 项目管理负责承载项目清单、项目识别、基础状态与进入动作。
  • Dashboard 负责承载项目级经营总览、指标汇总与子模块导航。

这两个模块共同解决的问题不是“把页面放出来”，而是让用户能够从项目入口进入到一个稳定、可识别、可继续操作的经营工作面。

2. 业务目标

  • 管理项目清单与项目基础状态
  • 支撑项目搜索、筛选、新建、导入、删除等动作
  • 在项目层形成统一经营入口
  • 汇总关键经营指标并连接各专业模块
  • 为场景、BOM、报价、分摊、跟踪等模块提供统一进入路径

3. 核心业务规则

3.1 项目是一级业务对象

所有场景、线束、BOM、报价、分摊、跟踪等能力都必须挂接在项目之下，而不是散落成平级对象。

3.2 项目列表负责入口，不替代专业模块

项目列表只负责识别、筛选、进入和基础状态呈现，不承载深度业务逻辑处理。

3.3 Dashboard 负责项目级汇总，不替代专业页面

Dashboard 是项目级总览和导航容器，不应替代 BOM、报价、设变、分摊、跟踪等专业页面。

3.4 项目状态应可支撑后续业务闭环

项目状态不应只是展示字段，而应能服务于后续场景推进、归档判断、经营状态识别。

4. 核心对象规格

4.1 项目对象

至少应包含：

  • projectId
  • projectCode
  • projectName
  • customerName
  • vehicleModel
  • projectStatus
  • harnessCount
  • createdAt
  • updatedAt

4.2 Dashboard 汇总对象

建议独立抽象汇总数据结构，至少包含：

  • projectId
  • summaryMetrics
  • costSummary
  • quoteSummary
  • profitSummary
  • warningSummary
  • versionSummary
  • updatedAt

5. 页面与交互要求

5.1 项目列表页面

至少支持：

  • 项目列表展示
  • 搜索（项目名 / 编号 / 客户）
  • 状态筛选
  • 新建
  • 导入
  • 删除
  • 进入项目详情

5.2 Dashboard 页面

至少支持：

  • 项目关键指标展示
  • 模块导航入口
  • 当前项目经营摘要
  • 场景、BOM、报价、Simulation、年降等主要入口

5.3 汇总信息要求

项目层至少应可看到：

  • 项目状态
  • 线束数量
  • 成本与报价摘要
  • 利润摘要
  • 关键异常或预警摘要预留

6. 数据与计算要求

6.1 项目列表数据要求

项目列表应基于真实项目对象与动态汇总结果呈现，而不是静态演示数据。

6.2 Dashboard 数据要求

Dashboard 应优先读取统一业务对象的汇总结果，而不是在页面内重复拼凑各模块私有逻辑。

6.3 汇总边界

Dashboard 可汇总结果，但不负责取代各专业模块的明细与专业计算。

7. 模块边界

  • 项目管理负责入口层，不负责深度业务计算。
  • Dashboard 负责项目级汇总与导航，不负责明细编辑。
  • 专业规则仍在 BOM、报价、分摊、设变、跟踪、预警等模块中承载。

8. 待确认项

  • 1. 项目状态枚举是否继续细化
  • 2. 项目层是否需要锁定 / 审批状态
  • 3. Dashboard 是否首期承载版本摘要或预警摘要
  • 4. Dashboard 是否支持场景切换总览

9. 开发实施建议

  • 1. 保持项目对象稳定
  • 2. 统一 Dashboard 汇总数据来源
  • 3. 增强项目状态对后续闭环的支撑能力
  • 4. 最后再深化管理摘要和异常摘要视图

10. 验收建议

A 级验收

  • 项目列表基础功能可用
  • Dashboard 可进入、可展示关键指标
  • 模块入口联通

B 级验收

  • Dashboard 汇总数据来源稳定
  • 项目状态能支持后续模块判断
  • 可承载预警 / 版本 / 场景摘要预留

C 级验收

  • 更重的管理视图整合
  • 项目层审批或锁定机制

11. 结论

项目管理与 Dashboard 是整个平台的门面层与工作台入口层。它们的价值不在于页面本身，而在于让用户围绕“项目”这个统一对象进入系统，并在项目级视角下连接后续全部经营闭环能力。

功能规格-场景管理

适用范围：高压线束精算与决策引擎

文档目标：把“一个项目多场景模型”从 PRD 级表述下钻为可供研发拆解的专项功能规格
当前判断：方向已明确，但主应用中仍属于【已确定未完全实现】

交叉引用

  • 对应总纲章节：2. 场景管理主线
  • 对应开发任务书：04-开发任务书/T02-场景管理.md
  • 对应测试清单：05-测试验收/QA01-P0主链路.md

1. 功能定位

场景管理不是附加页面，而是系统承载项目生命周期差异的核心业务容器。

如果没有场景管理，系统只能停留在“某个时间点算一次”的静态核算模式；有了场景管理，系统才能把初始报价、定点、设变、年降等不同阶段的经营事实放在同一项目下持续管理、比较、追溯和演进。

因此，场景管理的本质不是“多建几个页签”，而是建立：

  • 同一项目下多个经营事实容器
  • 场景之间可区分、可比较、可追溯的关系
  • 场景与 BOM、报价、分摊、跟踪、版本之间的稳定挂载关系

2. 业务目标

场景管理需要支撑以下业务目标：

  • 在同一项目内承载不同阶段的独立经营事实
  • 区分初始报价、定点、设变、年降等不同业务语义
  • 允许业务比较不同场景下的成本、报价、利润与回收状态
  • 保留每个场景的关键参数快照，而不是被后续修改覆盖
  • 为版本管理、设变管理、分摊回收、年降跟踪提供统一挂载基础

3. 场景定义

3.1 场景是什么

场景是项目下的一级业务容器，用于承载某一经营阶段下的完整业务事实。

3.2 场景不是什么

场景不是：

  • 页面标签的纯 UI 分组
  • 临时模拟结果的随意保存点
  • 任意复制项目形成的弱关联副本

3.3 场景类型

当前统一按以下四类场景定义：

  • 初始报价
  • 定点
  • 设变
  • 年降

其中：

  • 初始报价：用于项目早期报价测算与客户初步口径输出
  • 定点：用于定点后的相对稳定经营基线
  • 设变：用于承载 BOM 变更、替换、新增、取消等变更后的独立经营事实
  • 年降：用于承载周期性降价或成本改善影响下的经营事实

4. 核心业务规则

4.1 一个项目可包含多个场景

同一项目下允许存在多个场景，场景之间互相独立，但共享项目主身份。

4.2 场景必须有业务语义

每个场景都必须明确属于哪一种业务类型，而不是自由命名后无规则约束。

4.3 场景必须保留关键事实快照

每个场景都应保留至少以下关键事实：

  • 生命周期假设
  • 产量
  • 装车比
  • 费率快照
  • BOM 引用或 BOM 版本
  • 报价参数快照
  • 分摊状态
  • 版本状态

4.4 场景之间允许比较，但不能互相覆盖

场景之间允许对比，但一个场景的修改不应直接覆盖另一个场景的业务事实。

4.5 场景不是版本的替代物

场景与版本是两层不同概念：

  • 场景回答“当前处于什么经营语义与阶段”
  • 版本回答“同一场景或同一对象在时间线上的发布与追溯状态”

4.6 待确认规则不得提前写死

场景继承机制、冻结机制、发布机制在未拍板前，只能做结构预留，不能直接写死到默认逻辑中。

5. 场景对象规格

5.1 最低字段要求

场景对象至少应包含：

  • scenarioId：场景唯一标识
  • projectId：所属项目标识
  • scenarioType：场景类型（初始报价 / 定点 / 设变 / 年降）
  • scenarioName：场景显示名称
  • status：场景状态
  • lifecycleAssumption：生命周期假设
  • volume：产量
  • installRatio：装车比
  • rateSnapshotRef 或等价结构：费率快照引用
  • bomRef 或等价结构：BOM 引用
  • quoteParamSnapshotRef 或等价结构：报价参数快照引用
  • allocationStatus：分摊状态
  • versionRef 或等价结构：版本关联
  • createdAt
  • updatedAt
  • createdBy

5.2 推荐补充字段

建议预留：

  • sourceScenarioId：来源场景
  • compareBaselineScenarioId：默认比较基线场景
  • notes：备注
  • frozenAt：冻结时间
  • releasedAt：发布时间

5.3 状态建议

首期可采用简化状态：

  • 草稿
  • 计算中 / 调整中
  • 已冻结
  • 已发布
  • 已归档
说明：具体状态流是否全部首期实现，可分阶段推进；但数据结构建议先留位。

6. 场景与其他对象的关系

6.1 与项目关系

  • 一个项目包含多个场景
  • 场景必须归属于唯一项目

6.2 与线束号关系

  • 一个场景可包含多个线束号
  • 同一线束号在不同场景下允许出现不同结果

6.3 与 BOM 关系

  • 场景必须关联 BOM 数据来源
  • 不同场景可引用不同 BOM 版本或不同 BOM 快照

6.4 与报价关系

  • 场景必须关联客户报价参数或报价结果
  • 场景切换时不能丢失报价口径差异

6.5 与分摊关系

  • 场景必须承载一次性费用分摊状态
  • 不同场景的分摊回收结果应允许独立判断

6.6 与跟踪关系

  • 协议价落实、回收差异、异常处理等跟踪事项，应可挂载到具体场景

6.7 与版本关系

  • 场景不是版本，但场景可关联多个版本记录
  • 至少应能标记当前有效版本或当前引用版本

7. 功能范围

7.1 首期必须具备的能力

  • 场景列表展示
  • 新建场景
  • 查看场景基础信息
  • 进入场景后查看其关联的 BOM / 报价 / 核算结果
  • 场景之间基础比较入口
  • 场景与项目、版本的基础关联

7.2 第二阶段建议补齐的能力

  • 场景复制 / 继承
  • 场景冻结
  • 场景发布
  • 场景差异对比
  • 场景变更轨迹

7.3 暂不建议首期强推的能力

  • 过重审批流
  • 复杂多级权限矩阵
  • 大量自由配置的场景模板系统

8. 关键页面与交互要求

8.1 项目内场景入口

在项目详情中应存在稳定的场景入口，至少让用户看到：

  • 当前项目有哪些场景
  • 各场景类型
  • 各场景状态
  • 最近更新时间
  • 是否为当前主要经营基线

8.2 场景列表

场景列表至少应展示：

  • 场景名称
  • 场景类型
  • 状态
  • 产量 / 生命周期摘要
  • 关联版本信息
  • 最近更新时间

8.3 场景详情

场景详情不是单独重复 Dashboard，而应重点承载：

  • 场景基础信息
  • 与 BOM / Quote / Alloc / Tracking 的联动入口
  • 场景关键指标摘要
  • 当前版本状态
  • 比较入口

8.4 场景新建

首期新建场景至少应要求选择：

  • 场景类型
  • 场景名称
  • 所属项目

对于是否继承来源场景数据，当前保留为待确认项，不直接写死。

8.5 场景比较

比较能力至少应支持：

  • 成本比较
  • 报价比较
  • 利润比较
  • 分摊回收状态比较

9. 数据与计算要求

9.1 快照优先原则

场景一旦形成正式业务事实，其关键参数不应仅依赖全局设置实时回读，而应有快照或等价引用机制。

9.2 计算边界

场景管理本身不负责全部计算，但必须为计算引擎提供稳定上下文，包括：

  • 当前场景类型
  • 当前场景参数快照
  • 当前 BOM 引用
  • 当前费率快照
  • 当前版本状态

9.3 比较边界

比较结果应基于明确的场景基线，不应默认用“上一次打开的场景”之类隐式状态做比较。

10. 与其他模块的边界

10.1 与 Dashboard 的边界

  • Dashboard 做项目级汇总与导航
  • 场景管理做场景级业务容器与切换

10.2 与 Simulation 的边界

  • Simulation 负责假设测算
  • 场景管理负责正式经营场景承载
  • 模拟结果是否可转为场景，当前待确认

10.3 与版本管理的边界

  • 版本是追溯与发布维度
  • 场景是经营事实容器维度
  • 二者必须分层实现，不能混成一个对象

10.4 与设变 / 年降的边界

  • 设变、年降本身可表现为场景类型
  • 但其专业规则仍应保留在各自模块中

11. 待确认项

以下内容在场景管理专项规格中仍保留待确认：

  • 1. 新建场景时的继承机制
  • 继承哪些字段
  • 哪些字段必须重新确认
  • 2. 场景冻结 / 发布是否首期进入
  • 是否必须经过审批
  • 是否允许回退
  • 3. 模拟结果是否允许直接转场景
  • 4. 是否允许同类型多个场景并行存在
  • 例如多个设变场景、多个年降场景
  • 5. 场景默认比较基线的确定规则

12. 开发实施建议

12.1 建议研发顺序

  • 1. 先定义场景对象与关系结构
  • 2. 再补项目内场景入口与列表
  • 3. 再打通场景与 BOM / Quote / Alloc / Tracking 的引用链路
  • 4. 再补场景比较
  • 5. 最后再决定是否进入冻结 / 发布 / 继承等增强能力

12.2 首期实现建议

首期不求把场景能力一次做满，但至少要形成：

  • 可创建
  • 可识别类型
  • 可承载关键事实
  • 可挂接后续模块
  • 可比较基础结果

如果连这五点都不成立，系统仍然不算真正进入“一个项目多场景模型”。

13. 验收建议

A 级验收

  • 能在项目下看到多个场景
  • 能区分场景类型
  • 能进入具体场景查看其关键事实

B 级验收

  • 场景能挂接 BOM、报价、分摊、跟踪模块
  • 场景之间能做基础指标比较
  • 场景修改不会互相覆盖

C 级验收

  • 场景冻结 / 发布 / 继承 / 默认比较基线等增强能力
  • 若未拍板，可不纳入当前阶段必验范围

14. 结论

场景管理是统一 PRD 从“单次核算”走向“项目全生命周期经营管理”的关键拐点。它的价值不在于页面数量增加，而在于让系统第一次真正具备承载多个经营事实、比较多个经营状态、追溯多个经营阶段的能力。

因此，场景管理应被视为后续研发中的一级主线模块，而不是附属增强项。

1. 功能定位

BOM 工作簿是系统的核心主数据承载模块之一，负责把线束号、物料、数量、价格来源、分类规则、材料属性等信息组织成可计算、可编辑、可追溯的核算基础。

它不是普通表格页，而是内部成本、客户报价、设变分析、分摊回收、跟踪管理的共同上游。

2. 业务目标

  • 承载逐行 BOM 数据
  • 支撑线束号级与 BOM 行级分析
  • 形成成本核算与报价的主数据基础
  • 支撑 BOM 变更与版本比较
  • 支撑材料属性、金属属性、分类规则等业务映射

3. 核心业务规则

3.1 颗粒度必须到线束号 / BOM 行

不得回退为项目平均值或套级笼统汇总。

3.2 BOM 是多模块共同上游

3.3 BOM 行必须有稳定业务属性

  • 金属属性
  • 所属线束号

3.4 BOM 变更必须可追溯

BOM 不应只有当前结果，还应支持差异识别、版本关系或快照引用。

6. 数据与计算要求

7. 模块边界

  • BOM 工作簿负责主数据承载，不直接替代报价、设变、分摊、跟踪等专业模块。
  • BOM 工作簿负责数据基础与编辑，不承担管理决策展示职责。
  • BOM 工作簿可提供差异基础，但设变事件仍应在设变模块中承载。

  • 2. 分类规则与模板字段映射深度
  • 3. 价格来源是否需要更细颗粒类型治理

BOM 工作簿是整个系统最重要的上游基础之一。它的价值不只是“把数据列出来”，而是为内部成本、客户报价、设变分析和经营闭环提供统一、稳定、可追溯的数据事实基础。

1. 功能定位

客户报价工作台是系统双引擎中的客户侧结果承载模块，负责把内部核算能力转化为面向客户的报价输出与商务判断依据。

它不是简单的导出页面，也不是内部成本结果的原样镜像，而是一个承载客户口径参数、报价结果、利润空间判断、模板输出映射的业务工作台。

对于吉利客户，真实客户最终报价单虽然可作为业务参照，但系统主对象仍然是内部核算结果。客户最终报价单是销售按商务策略调整后提交给客户并被承认的版本，系统只需关注其中与内部核算对照最关键的口径，如分摊费用及回收方式、出厂价、到厂价和定点状态，不应要求完整映射销售最终报价结构。

2. 业务目标

客户报价工作台需要支撑以下目标：

  • 形成客户口径下的报价结果
  • 区分内部实绩成本与客户报价口径
  • 支撑销售进行商务判断与客户报价输出
  • 支撑与内部真实成本进行差异比较
  • 支撑模板映射与对外输出

3. 业务范围

3.1 报价对象范围

至少覆盖：

  • 项目
  • 场景
  • 线束号
  • 报价版本或报价快照

3.2 报价内容范围

至少覆盖：

  • 材料相关报价结果
  • 人工 / 制造 / 管理等口径映射
  • 利润要求
  • 客户模板字段输出
  • 与内部成本差异结果

对于吉利客户，系统在输出或对照层面优先关注：

  • 出厂价
  • 到厂价
  • 分摊费用表达
  • 回收方式表达
  • 是否已定点/已被客户承认

4. 核心业务规则

4.1 客户报价与内部实绩成本必须分层

客户报价结果不能直接替代内部真实成本结果，二者必须并行存在。

4.2 报价必须保留客户口径参数

客户报价结果必须能追溯到当时采用的参数口径，而不是后续被全局配置覆盖。

4.3 报价结果必须可比较

系统至少应支持比较：

  • 客户报价 vs 内部实绩成本
  • 场景 A 报价 vs 场景 B 报价
  • 当前报价 vs 历史报价版本

4.4 模板输出不应主导内核结构

客户模板适配属于输出层，不能反向决定核心业务对象与计算结构。

4.5 参数权限边界必须可治理

销售可调、财务锁定、需审批项的边界在未拍板前不能写死，但系统结构应预留治理能力。

5. 核心对象规格

5.1 报价快照对象

至少应包含：

  • quoteSnapshotId
  • projectId
  • scenarioId
  • harnessId
  • quoteParamsRef
  • customerTemplateType
  • quoteResult
  • internalCostBaseline
  • profitGap
  • status
  • createdAt
  • updatedAt
  • createdBy

如果客户为吉利，则 quoteResult 不应理解为销售最终提交客户的整套报价单镜像，而应主要承载系统内部可稳定管理和比较的客户口径结果，并允许补充以下对照字段：

  • exWorksPriceRef：出厂价对照口径
  • arrivalPriceRef：到厂价对照口径
  • allocationExpressionRef：分摊费用表达参考
  • recoveryExpressionRef：回收方式表达参考
  • customerAcceptedStatus：是否已定点/已被客户承认

5.2 报价参数对象

建议独立维护报价参数快照，至少应包含：

  • quoteParamSnapshotId
  • projectId
  • scenarioId
  • parameterSet
  • lockedFields
  • editableFields
  • approvalRequiredFields
  • createdAt

6. 页面与交互要求

6.1 报价主页面

至少展示：

  • 场景 / 线束号
  • 客户报价结果
  • 内部成本基线
  • 差异值
  • 利润空间
  • 输出状态

6.2 参数区

至少支持：

  • 查看当前报价参数
  • 区分可调 / 锁定 / 待审批字段
  • 查看参数来源或快照信息

6.3 模板输出区

至少支持：

  • 查看当前模板类型
  • 查看映射结果预览
  • 导出或输出入口预留

对于吉利客户，预览区或对照区优先展示：

  • 出厂价对照
  • 到厂价对照
  • 分摊费用及回收方式说明
  • 是否已定点/客户已承认

7. 数据与计算要求

7.1 输入基础

报价结果至少依赖：

  • 当前场景
  • 当前线束号或 BOM 汇总结果
  • 报价参数快照
  • 内部成本基线
  • 模板类型

7.2 快照原则

报价参数与报价结果必须可追溯，不能只依赖实时全局配置。

7.3 差异计算

至少应能稳定计算：

  • 报价 - 内部成本
  • 报价毛利空间
  • 不同场景 / 不同版本报价差异

8. 模块边界

  • 报价工作台负责客户口径结果，不替代内部实绩成本引擎。
  • 报价工作台负责输出与比较，不替代 Settings 中的参数治理。
  • 报价工作台可引用 BOM、场景、版本结果，但不负责维护它们的主数据。
  • 吉利客户最终报价单是业务对照结果，不是系统正式计算内核；系统仍以内部报价核算结构为主。

  • 正式发布与外部协同流程

客户报价工作台是双引擎中面向客户与商务决策的一侧，它的价值不只在于出一个报价数字，而在于让系统能同时看到内部真实成本与客户输出口径，从而支撑调价、谈判、利润空间判断与客户模板交付。

Simulation 与年降管理共同承担“面向未来变化的经营判断”能力，但两者的业务语义不同：

  • Simulation 负责假设测算、敏感度观察与决策预演

2. 业务目标

  • 支撑关键参数假设调整
  • 观察成本、报价、利润对关键变量变化的响应
  • 支撑金属联动、调价空间、情景比较
  • 支撑年降对成本、报价、利润的周期影响分析
  • 为经营判断提供预演基础

3. 核心业务规则

3.1 Simulation 属于假设测算，不等于正式版本

Simulation 结果可以作为判断依据，但不应默认等价为正式经营事实。

3.2 年降必须有明确业务语义

年降不应被实现为简单的数字减值，而应是有周期、阶段与影响链路的业务对象。

3.3 仿真与正式结果必须分层

仿真结果、正式报价结果、正式成本结果应保持区分，避免互相覆盖。

3.4 年降与场景、版本应可挂接

年降结果至少应能关联到具体场景和版本上下文。

  • simulationId
  • projectId
  • scenarioId
  • inputSnapshotRef
  • adjustedParameters
  • resultSummary
  • createdAt
  • createdBy

4.2 年降对象

  • annualDropId
  • projectId
  • scenarioId
  • period
  • dropRate
  • affectedScope
  • resultSummary
  • status
  • createdAt
  • updatedAt

5.1 Simulation 页面

  • 查看关键参数
  • 调整关键变量
  • 查看结果变化
  • 场景或基线结果对照

5.2 年降页面

至少支持：

  • 查看年降参数
  • 查看周期影响结果
  • 查看成本 / 报价 / 利润变化
  • 与场景或版本结果建立对照

6.1 仿真输入要求

  • 当前场景
  • 当前参数快照
  • 当前 BOM / 成本 / 报价基线
  • 被调整的关键变量

6.2 年降输入要求

至少依赖：

  • 周期
  • 年降率
  • 影响范围
  • 当前成本 / 报价基线

6.3 快照原则

8. 待确认项

  • 1. 仿真结果是否允许一键转场景
  • 2. 年降是否视为独立场景还是场景内状态
  • 3. 年降版本是否首期纳入正式版本治理
  • 4. 仿真结果保存粒度与保留期限

9. 开发实施建议

  • 1. 先明确仿真与正式事实的边界
  • 2. 再定义仿真任务与年降对象
  • 3. 再打通与场景、版本、报价、成本结果的引用关系
  • 4. 最后增强保存、比较和转化能力

  • 年降能和场景 / 版本稳定挂接
  • 能做基础比较

C 级验收

  • 仿真转场景
  • 年降治理深化
  • 更复杂的周期序列分析

11. 结论

Simulation 与年降管理的价值，在于让系统不只会呈现当前结果，还能支撑未来变化判断。它们是经营决策层的重要增强能力，但必须与正式场景、版本和报价结果保持清晰边界。

F06-分摊回收跟踪

功能规格-分摊回收跟踪

适用范围：高压线束精算与决策引擎
来源依据：PRD_统一版_高压线束精算与决策引擎.md、PRD_统一版_研发实施清单.md
文档目标：把“一次性费用分摊 + 回收跟踪”从 PRD 级表述下钻为可供研发拆解的专项功能规格
当前判断：方向已明确，但主应用中仍属于【已确定未完全实现】

交叉引用

  • 对应总纲章节：6. 分摊回收跟踪
  • 对应开发任务书：04-开发任务书/T06-分摊回收跟踪.md
  • 对应测试清单：05-测试验收/QA02-P1执行闭环.md
  • 相关数据结构说明书：06-数据结构说明书/DS01-核算表数据结构与计算逻辑说明书.md

1. 功能定位

分摊回收跟踪不是一个单纯的成本分配界面，而是系统将一次性费用从“发生即消失的成本项”转为“可持续回收、可持续跟踪的经营对象”的关键能力。

如果没有分摊回收跟踪，系统只能在核算时把工装、模具、试验、研发等一次性费用静态摊入成本；有了分摊回收跟踪，系统才能持续判断：

  • 某笔一次性费用已经回收多少
  • 还有多少未回收
  • 回收是否偏离预期
  • 回收完成后是否进入调价判断

因此，该模块的本质不是“分摊一下”，而是建立一次性费用的完整经营闭环。

2. 业务目标

分摊回收跟踪需要支撑以下目标：

  • 支撑一次性费用按线束号独立、按根分摊
  • 支撑分摊结果挂接到具体场景、线束号与项目
  • 支撑按装车比 × 累计产量持续跟踪回收进度
  • 支撑已回收 / 未回收 / 已完成等状态识别
  • 支撑回收完成后的调价提醒或后续经营动作
  • 支撑管理层、财务、销售查看回收进度与差异

3. 业务范围

3.1 一次性费用范围

当前至少包括：

  • 工装费
  • 模具费
  • 试验费
  • 研发费
  • 其他需要按生命周期回收的非经常性投入

3.2 分摊对象范围

分摊必须至少支持挂接到：

  • 项目
  • 场景
  • 线束号

3.3 回收跟踪范围

回收跟踪至少要覆盖：

  • 计划应回收金额
  • 实际已回收金额
  • 未回收金额
  • 回收进度比例
  • 回收完成状态

4. 核心业务规则

4.1 按线束号独立分摊

一次性费用必须按线束号独立分摊，不能回退为按项目整体或按套笼统平均分摊。

4.2 按根分摊

分摊口径应以线束号产出单位为基础，按根分摊。

4.3 回收进度按装车比 × 累计产量判断

分摊回收跟踪的核心依据是装车比与累计产量，而不是仅依据计划表静态推算。

4.4 分摊与回收是两层概念

  • 分摊回答“这笔一次性费用应该如何进入成本口径”
  • 回收回答“这笔一次性费用在执行过程中已经收回多少”

二者必须分层实现，不能混成一个字段。

4.5 回收完成后应触发经营提醒

当某笔一次性费用回收完成时，系统至少应具备提醒能力，用于后续调价或经营判断。

4.6 不同场景下的分摊状态允许独立

不同场景下，同一线束号或同一费用项允许存在不同的分摊与回收判断结果。

5. 核心对象规格

5.1 分摊项对象

分摊项至少应包含：

  • allocationItemId
  • projectId
  • scenarioId
  • harnessId 或等价线束号标识
  • expenseType：费用类型
  • expenseName
  • totalAmount：总金额
  • allocationBasis：分摊依据
  • unitAllocationAmount：单根分摊金额
  • plannedRecoveryAmount：计划回收金额
  • actualRecoveredAmount：已回收金额
  • remainingRecoveryAmount：未回收金额
  • recoveryProgress：回收进度
  • status：分摊 / 回收状态
  • createdAt
  • updatedAt

5.2 回收记录对象

建议独立建立回收记录，用于承载执行轨迹。至少应包含：

  • recoveryRecordId
  • allocationItemId
  • projectId
  • scenarioId
  • period 或时间点
  • cumulativeVolume
  • installRatioSnapshot
  • recoveredAmount
  • remainingAmount
  • status
  • remark
  • createdAt

5.3 推荐补充字段

建议预留：

  • sourceVersionId
  • baselineVolume
  • targetRecoveryDate
  • completedAt
  • priceAdjustmentReminderStatus

6. 状态设计建议

6.1 分摊项状态

首期可采用：

  • 待分摊
  • 已分摊待回收
  • 回收中
  • 已回收完成
  • 已关闭

6.2 回收异常状态

建议预留：

  • 回收滞后
  • 数据异常
  • 已超额回收
  • 待复核
说明：异常状态是否首期页面化完整实现可后置，但数据结构建议先预留。

7. 与其他对象和模块的关系

7.1 与场景关系

  • 分摊项必须归属于具体场景
  • 场景切换时，分摊状态不得丢失

7.2 与线束号关系

  • 分摊以线束号为最小主要承载对象之一
  • 同一项目下不同线束号的分摊不应互相平均覆盖

7.3 与报价关系

  • 分摊结果会影响内部成本口径
  • 回收完成后可能影响客户调价判断
  • 对吉利客户，系统中的回收完成判断仍应立足内部核算与内部报价结果；如需对照客户最终报价单，仅参考分摊费用及回收方式、出厂价、到厂价和定点状态

7.4 与跟踪关系

  • 回收进度、回收差异、完成提醒本质上也属于跟踪事件来源
  • 跟踪模块应可引用分摊回收结果

7.5 与预警关系

  • 回收滞后、异常、超支等可成为预警触发条件

7.6 与版本关系

  • 分摊规则、分摊快照与回收状态应支持版本追溯

8. 功能范围

8.1 首期必须具备的能力

  • 分摊项列表
  • 新增 / 查看分摊项
  • 查看分摊总额、单根分摊金额
  • 查看计划回收、已回收、未回收
  • 查看当前状态
  • 回收完成提醒入口

8.2 第二阶段建议补齐的能力

  • 回收记录明细
  • 回收差异分析
  • 回收滞后预警
  • 场景间分摊差异比较
  • 与跟踪、预警、报价模块的联动闭环

8.3 暂不建议首期做重的能力

  • 过度复杂的自动审批流
  • 过度复杂的财务多账套映射
  • 与主经营闭环无直接关系的超细颗粒台账功能

9. 页面与交互要求

9.1 分摊管理页面

分摊管理页面至少应展示：

  • 费用类型
  • 线束号
  • 总金额
  • 单根分摊金额
  • 计划回收金额
  • 已回收金额
  • 未回收金额
  • 当前状态

9.2 回收进度展示

回收进度至少应支持：

  • 百分比
  • 已回收 / 未回收金额
  • 状态标识
  • 最近更新时间

9.3 分摊详情

详情页至少应承载：

  • 分摊依据
  • 当前场景
  • 线束号
  • 关键参数快照
  • 回收进度轨迹
  • 是否触发调价提醒

9.4 调价提醒

当回收完成时，至少需要有明确提醒入口，而不能只靠用户自己推断。

10. 数据与计算要求

10.1 计算基础

分摊回收至少依赖以下输入：

  • 一次性费用总额
  • 线束号对象
  • 产量
  • 装车比
  • 累计产量
  • 当前场景上下文

10.2 快照原则

装车比、产量、费率等会影响回收判断的关键参数，应在关键节点保留快照或等价记录，以避免后续全局参数变化后回收结果不可解释。

10.3 计算边界

分摊回收模块负责：

  • 分摊金额形成
  • 回收进度判断
  • 状态识别
  • 提醒触发

但不负责替代报价引擎、预警系统或跟踪系统的全部职责。

11. 与其他模块的边界

11.1 与 BOM 的边界

  • BOM 提供成本承载基础与线束号关系
  • 分摊模块负责一次性费用逻辑，不负责 BOM 明细维护

11.2 与报价模块的边界

  • 报价模块负责客户口径结果
  • 分摊回收模块负责内部一次性费用成本与回收链路
  • 二者可联动，但职责不能混写

11.3 与跟踪模块的边界

  • 分摊回收模块负责形成回收事实
  • 跟踪模块负责汇总执行差异与业务事件

11.4 与预警模块的边界

  • 分摊模块提供状态与异常信号
  • 预警系统负责分级、触发和汇总

12. 待确认项

  • 1. 回收完成后的业务动作
  • 仅提醒
  • 进入调价流程
  • 进入审批流程
  • 2. 回收金额的实际认定来源
  • 是否直接依据累计产量推算
  • 是否需要人工确认或外部数据导入
  • 3. 是否允许手动修正回收状态
  • 4. 超额回收、回收冲销等特殊情形是否首期处理
  • 5. 多场景下同一费用项如何做比较与继承

13. 开发实施建议

13.1 建议研发顺序

  • 1. 先定义分摊项与回收记录对象
  • 2. 再打通场景、线束号、金额、产量、装车比的基础关系
  • 3. 再实现分摊结果和回收状态计算
  • 4. 再做页面展示与详情页
  • 5. 最后补提醒、预警、跟踪联动

13.2 首期实现建议

首期至少要形成：

  • 可记录一次性费用
  • 可绑定线束号和场景
  • 可计算单根分摊
  • 可判断回收进度
  • 可识别回收完成状态

如果这五点不能成立，系统仍然只是“静态分摊展示”，不算真正形成分摊回收闭环。

14. 验收建议

A 级验收

  • 能看到分摊项列表
  • 能看到总金额、单根分摊、已回收、未回收
  • 能区分状态

B 级验收

  • 能按装车比 × 累计产量判断回收进度
  • 能和场景、线束号稳定关联
  • 回收完成后能触发提醒

C 级验收

  • 滞后预警
  • 超额回收
  • 冲销处理
  • 审批化流转

若这些仍未拍板，可不纳入当前阶段强制验收范围。

15. 结论

分摊回收跟踪的价值，不在于把一次性费用“分进成本”这一瞬间，而在于把这类费用从静态核算项升级为可持续追踪、可持续解释、可持续决策的经营对象。

它是系统从“会算”走向“会管”的关键模块之一，应与场景管理、跟踪管理、预警系统一起，被视为经营闭环主链路，而不是可有可无的附属台账。

1. 功能定位

设变与跟踪不是两个彼此孤立的页面，而是一条连续的经营执行链路。

  • 设变回答：发生了什么变化，变化影响了哪些对象，成本与报价怎么变。
  • 跟踪回答：这些变化、差异和经营动作后续执行到了哪里，是否偏离预期，是否需要处理与关闭。

如果没有设变管理，系统无法把 BOM 新增、替换、取消等变化转成有结构的经营事实；如果没有跟踪管理，系统即使算出了变化影响，也无法持续管理协议价落实、差异消化、残余材料池处理和异常闭环。

因此，这一组模块的本质是把“变化”转成“可持续执行与可持续回溯的经营事件链”。

2. 业务目标

设变与跟踪需要支撑以下目标：

  • 记录并管理 BOM 变化、替换、新增、取消等设变事实
  • 识别设变对成本、报价、利润、残余材料池的影响
  • 支撑客户变更报价与内部经营判断
  • 跟踪协议价落实、执行差异、残余材料池处理与异常状态
  • 让设变结果、跟踪结果、版本状态与预警状态形成闭环

3. 业务范围

3.1 设变范围

当前至少覆盖：

  • BOM 新增
  • BOM 替换
  • BOM 取消
  • 数量变更
  • 材料或价格属性变化

3.2 跟踪范围

当前至少覆盖：

  • 协议价落实
  • 进度价差距跟踪
  • 分摊回收执行差异
  • 残余材料池 / 呆滞提报处理
  • 关键异常与关闭状态

4. 核心业务规则

4.1 设变必须以结构化业务事件存在

设变不能只是“改了 BOM 数据”这一结果，而应形成独立的设变事件对象，记录变更前、变更后及影响范围。

4.2 设变影响必须可解释

系统至少应能回答：

  • 哪些 BOM 行发生变化
  • 对哪些线束号产生影响
  • 成本变化多少
  • 报价变化多少
  • 是否形成残余材料池或呆滞提报对象

4.3 进度价本质是差距追踪

进度价不应被实现为加权混合价，而应被视为协议价与当前批量价之间的差距追踪结果。

4.4 残余材料池不继续计入当前产品成本

设变导致的取消料号、未匹配料号等形成的残余材料池，本质应进入呆滞提报流程，而不是继续分摊进当前产品成本。

4.5 跟踪对象必须有状态

所有需要持续跟进的对象，都应具有明确状态，而不是停留在备注或静态表格记录中。

4.6 跟踪必须支持关闭或归档

跟踪不是永远打开的流水账，后续至少应支持关闭、完成、归档等状态判断。

7.2 与场景的关系

  • 来源对象
  • 差异或问题摘要
  • 责任归属预留
  • 最近更新时间

9.4 跟踪详情

详情至少应承载：

  • 跟踪背景
  • 当前结果
  • 待执行动作
  • 实际处理结果
  • 关闭或归档状态

10. 数据与计算要求

10.1 设变计算基础

设变影响至少依赖：

  • 变更前 BOM
  • 变更后 BOM
  • 线束号关系
  • 成本计算上下文
  • 报价计算上下文
  • 当前场景与版本

10.2 跟踪计算基础

跟踪判断至少依赖：

  • 协议价与当前批量价
  • 分摊回收状态
  • 残余材料池状态
  • 异常阈值或业务规则

10.3 快照原则

设变前后差异、协议价差距、异常状态等关键内容应具备快照或稳定记录，避免后续被全局参数覆盖后无法追溯。

11. 与其他模块的边界

11.1 与 BOM 的边界

  • BOM 管理负责明细数据维护
  • 设变模块负责把 BOM 变化组织成业务事件

11.2 与报价模块的边界

  • 报价模块负责客户结果形成
  • 设变与跟踪模块负责识别变化与执行差距，不替代报价引擎本身

11.3 与分摊模块的边界

  • 分摊模块负责一次性费用与回收逻辑
  • 跟踪模块负责引用其状态形成执行闭环

11.4 与预警模块的边界

  • 跟踪提供异常信号
  • 预警系统负责分级与提醒承载

12. 待确认项

  • 1. 设变是否必须形成版本级差异报告
  • 2. 设变是否必须审批后生效
  • 3. 跟踪项是否允许人工关闭
  • 4. 跟踪项是否需要责任人和时限机制
  • 5. 进度价差距跟踪是否需要独立专题视图

13. 开发实施建议

13.1 建议研发顺序

  • 1. 先定义设变事件与跟踪项对象
  • 2. 再打通 BOM、场景、版本、报价、分摊之间的引用关系
  • 3. 再实现设变影响识别
  • 4. 再实现跟踪状态流
  • 5. 最后补关闭、预警、管理视图汇总

13.2 首期实现建议

首期至少要形成：

  • 可记录设变事件
  • 可查看变更影响
  • 可生成或挂接跟踪项
  • 可查看当前状态
  • 可区分未处理 / 处理中 / 已完成

如果这五点不成立，系统仍然只是“有变化但不可管理”的静态记录。

14. 验收建议

A 级验收

  • 能创建或查看设变事件
  • 能看到影响对象与影响结果
  • 能看到跟踪项列表和状态

B 级验收

  • 设变能和 BOM / 场景 / 版本稳定关联
  • 跟踪能覆盖协议价、进度价、分摊回收或残余材料至少几类核心对象
  • 跟踪状态能推进，不是静态展示

C 级验收

  • 审批流
  • 责任人和时限治理
  • 关闭机制细化
  • 版本差异专题报告

15. 结论

设变与跟踪的意义，在于让系统不只知道“数据变了”，而是知道“为什么变、影响了什么、后续处理到了哪里、经营风险有没有关闭”。

它是系统从“会核算”走向“会执行、会闭环”的关键主链路，应与场景管理、分摊回收、预警系统共同构成经营闭环的执行层骨架。

F08-预警系统与Alerts

功能规格-预警系统与 Alerts

适用范围：高压线束精算与决策引擎
来源依据：PRD_统一版_高压线束精算与决策引擎.md、PRD_统一版_研发实施清单.md
文档目标：把“预警系统 + Alerts 页面承载”从 PRD 级表述下钻为可供研发拆解的专项功能规格
当前判断：方向已明确，但主应用中仍属于【已确定未完全实现】

交叉引用

  • 对应总纲章节：8. 预警系统与Alerts
  • 对应开发任务书：04-开发任务书/T08-预警系统与Alerts.md
  • 对应测试清单：05-测试验收/QA02-P1执行闭环.md

1. 功能定位

预警系统与 Alerts 不是同一个概念，但二者必须组成一套闭环：

  • 预警系统负责规则、阈值、触发、状态与分级
  • Alerts 负责把预警结果页面化、可查看、可分类、可跟进

如果只有预警阈值配置入口，没有触发链路和查看入口，系统仍然只是“有参数、无预警”；如果只有 Alerts 页面，没有稳定规则来源，页面也只会成为静态信息看板。

因此，这一组能力的本质是把经营异常从“隐含在数据里的偏差”升级为“可识别、可分级、可跟进的经营提醒对象”。

2. 业务目标

预警系统与 Alerts 需要支撑以下目标：

  • 识别关键经营异常
  • 把异常按统一规则转成预警对象
  • 区分不同级别、不同类型、不同处理状态
  • 为财务、销售、管理层提供统一查看入口
  • 为跟踪、处理、关闭提供触发基础

3. 业务范围

3.1 预警类型范围

当前至少覆盖：

  • 金属价格预警
  • 分摊回收预警
  • 成本异常 / 超支预警
  • 关键执行节点预警

3.2 Alerts 页面范围

至少覆盖：

  • 预警列表
  • 分类查看
  • 级别展示
  • 状态展示
  • 跳转关联对象

4. 核心业务规则

4.1 预警必须来自明确规则

预警不能依赖临时人工判断或页面写死文案，而应来自统一的规则、阈值和触发条件。

4.2 预警必须挂接业务对象

每条预警至少应能关联到某个项目、场景、线束号、分摊项、跟踪项或版本对象。

4.3 预警必须具有级别或优先级

预警至少要具备可排序的严重程度判断，否则无法支持经营处理优先级。

4.4 预警系统与 Alerts 必须分层实现

  • 预警系统是规则层
  • Alerts 是展示与查看层

二者可以关联，但不能在概念与设计上混成一个页面需求。

4.5 预警不等于问题已关闭

预警触发只代表识别到异常，不代表问题已经被处理。后续是否关闭，应与跟踪或处理状态联动。

5. 核心对象规格

5.1 预警规则对象

预警规则至少应包含：

  • warningRuleId
  • warningType
  • name
  • thresholdConfig
  • triggerCondition
  • severityRule
  • scope
  • status
  • createdAt
  • updatedAt

5.2 预警实例对象

预警实例至少应包含：

  • warningId
  • projectId
  • scenarioId
  • warningType
  • severity
  • title
  • description
  • sourceRef
  • currentStatus
  • triggeredAt
  • updatedAt

5.3 推荐补充字段

建议预留：

  • assignedTo
  • closedAt
  • closeReason
  • trackingRef
  • versionRef
  • snapshotRef

6. 状态与级别设计建议

6.1 预警状态

首期可采用：

  • 新触发
  • 待处理
  • 处理中
  • 已关闭
  • 已忽略

6.2 严重级别

首期建议至少支持：

  • 高
  • 中
  • 低
说明：阈值、颜色、展示优先级目前仍属于待确认项，但级别结构建议先保留。

7. 与其他对象和模块的关系

7.1 与场景关系

  • 预警应可挂接具体场景
  • 同一项目不同场景的预警应独立判断

7.2 与分摊回收关系

  • 回收滞后、未完成、异常超支等可形成预警来源

7.3 与设变 / 跟踪关系

  • 超期未处理、差异未关闭、关键执行节点异常可形成预警来源
  • 预警关闭可与跟踪项状态联动

7.4 与管理决策舱关系

  • 管理视图可汇总预警结果
  • 但不应替代 Alerts 的详细查看能力

7.5 与系统设置关系

  • 阈值配置可在 Settings 中维护
  • 但规则触发与实例生成不应停留在 Settings 页面本身

8. 功能范围

8.1 首期必须具备的能力

  • 预警规则基础配置入口
  • 预警实例生成能力
  • Alerts 列表页
  • 级别、类型、状态展示
  • 关联对象跳转入口

8.2 第二阶段建议补齐的能力

  • 预警筛选与分类
  • 预警详情页
  • 预警关闭与复核机制
  • 预警与跟踪模块联动
  • 管理视图汇总看板

8.3 暂不建议首期强推的能力

  • 复杂消息通知矩阵
  • 过度细化的多级升级流转
  • 与主闭环无直接关系的通用消息中心化建设

9. 页面与交互要求

9.1 Alerts 列表页

至少应展示：

  • 预警标题
  • 预警类型
  • 严重级别
  • 当前状态
  • 所属项目 / 场景
  • 最近更新时间

9.2 预警详情

详情至少应承载：

  • 触发原因
  • 规则来源
  • 关联对象
  • 当前状态
  • 处理建议或处理记录预留

9.3 分类筛选

首期至少建议支持：

  • 按类型筛选
  • 按级别筛选
  • 按状态筛选
  • 按项目 / 场景筛选

9.4 跳转联动

从 Alerts 中至少应能跳转到：

  • 对应项目
  • 对应场景
  • 对应分摊 / 跟踪 / 报价等来源对象

10. 数据与计算要求

10.1 触发基础

预警触发至少依赖：

  • 阈值配置
  • 业务对象当前状态
  • 规则判断逻辑
  • 对象上下文（项目 / 场景 / 类型）

10.2 快照原则

预警实例应记录触发时的上下文快照或等价说明，避免后续业务对象变化后无法解释为什么当时触发。

10.3 计算边界

预警系统负责：

  • 规则配置
  • 条件判断
  • 实例生成
  • 级别与状态维护

Alerts 负责：

  • 页面承载
  • 分类查看
  • 详情查看
  • 关联对象跳转

11. 与其他模块的边界

11.1 与 Settings 的边界

  • Settings 负责配置入口
  • 预警系统负责真正的触发规则与实例生成

11.2 与 Tracking 的边界

  • Tracking 负责问题执行闭环
  • 预警负责发现异常并提示进入处理链路

11.3 与 Dashboard / ManagerDashboard 的边界

  • Dashboard / ManagerDashboard 可展示预警摘要
  • Alerts 负责详细预警查看与分类入口

12. 待确认项

  • 1. 预警分级标准
  • 阈值
  • 颜色
  • 严重程度算法
  • 2. 预警关闭机制
  • 是否允许人工关闭
  • 是否必须关联跟踪项关闭
  • 3. 是否需要责任人和处理时限
  • 4. 是否需要外部通知能力
  • 飞书
  • 站内消息
  • 其他协同方式
  • 5. Alerts 是否首期支持批量操作

13. 开发实施建议

13.1 建议研发顺序

  • 1. 先定义预警规则对象与预警实例对象
  • 2. 再明确与分摊、跟踪、场景、版本的来源关系
  • 3. 再实现规则触发与实例生成
  • 4. 再实现 Alerts 列表与详情
  • 5. 最后补关闭、复核、管理视图汇总

13.2 首期实现建议

首期至少要形成：

  • 有配置入口
  • 有规则触发
  • 有实例生成
  • 有列表查看
  • 有对象跳转

如果这五点不成立，系统仍然不算真正具备预警闭环。

14. 验收建议

A 级验收

  • 能产生预警实例
  • 能看到 Alerts 列表
  • 能看到类型、级别、状态

B 级验收

  • 预警能与项目 / 场景 / 来源对象稳定关联
  • 能按类型、级别、状态筛选
  • 能跳转到关联对象

C 级验收

  • 关闭机制
  • 责任人和时限治理
  • 外部通知
  • 管理视图聚合深化

15. 结论

预警系统与 Alerts 的价值，不在于多一页提醒列表，而在于把经营异常从“埋在结果里的偏差”转成“可被系统主动识别、主动暴露、可被后续处理闭环接住的管理对象”。

它是系统从“能算、能看”进一步走向“能提醒、能治理”的关键能力，应作为治理闭环主线模块持续推进。

1. 功能定位

管理决策舱是系统面向管理层的结果汇总与经营判断入口，不负责逐行编辑，不替代执行型页面，而是把多模块结果聚合成适合经营决策的视图。

2. 业务目标

  • 汇总利润、成本、报价、差异、回收进度与异常趋势
  • 提供跨项目、跨场景或跨版本的管理视角
  • 暴露关键异常与经营拐点
  • 支撑管理层快速判断而非逐页钻取

3. 核心业务规则

3.1 决策舱展示的是汇总结果，不是业务明细编辑页

3.2 决策舱的数据来源必须来自统一业务对象，而不是页面各自临时计算

3.3 决策舱应优先暴露差异、趋势、异常，而不是堆叠明细字段

3.4 决策舱不替代专业模块，必须保留跳转关系

4. 核心对象规格

建议抽象管理汇总对象，至少包含：

  • dashboardSummaryId
  • scope
  • costSummary
  • quoteSummary
  • profitSummary
  • warningSummary
  • allocationSummary
  • trackingSummary
  • versionSummary
  • updatedAt

5. 页面与交互要求

5.1 首页摘要区

至少应展示：

  • 关键利润指标
  • 成本 / 报价差异摘要
  • 回收进度摘要
  • 预警摘要

5.2 对比区

建议支持：

  • 场景对比
  • 项目对比
  • 版本对比

5.3 跳转要求

每一类摘要都应能跳转回对应专业模块查看原因与明细。

6. 数据与计算要求

  • 汇总数据必须来自稳定来源
  • 异常、预警、跟踪、分摊结果必须可被统一聚合
  • 汇总口径需与主业务模块保持一致

7. 模块边界

  • 管理决策舱负责汇总判断，不负责编辑明细。
  • 具体规则仍在下游模块承载。
  • 决策舱可以做管理表达，但不应复制一套独立业务逻辑。

8. 待确认项

  • 1. 是否首期支持跨项目对比
  • 2. 是否首期支持版本治理摘要
  • 3. 是否纳入异常关闭与审计摘要
  • 4. 是否需要独立管理角色视图配置

9. 开发实施建议

  • 1. 先统一汇总对象结构
  • 2. 再接入成本、报价、分摊、预警、跟踪摘要
  • 3. 最后深化对比能力与管理层表达样式

10. 验收建议

A 级验收

  • 能展示关键管理摘要
  • 能跳转到专业模块

B 级验收

  • 能稳定汇总多个核心模块结果
  • 能支撑趋势或差异表达

C 级验收

  • 跨项目对比
  • 治理摘要
  • 更重的管理驾驶舱能力

11. 结论

管理决策舱的价值，在于让系统从业务执行工具进一步升级为经营判断工具，把分散在项目、场景、报价、分摊、跟踪、预警中的结果压缩成管理层真正关心的差异、趋势与异常。

1. 功能定位

版本与发布治理是系统从“可计算”走向“可管理、可追溯、可审计”的关键基础能力。

它回答的不是某个结果怎么算，而是：

  • 哪个版本当前有效
  • 谁发布了这份基准或结果
  • 是否经过审核
  • 后续如何追溯和审计

2. 业务目标

  • 支撑费率、BOM、场景、报价等对象的版本化管理
  • 支撑关键结果发布与追溯
  • 支撑基准发布流
  • 支撑审计标识与状态留痕
  • 为权限隔离和异常关闭提供治理基础

3. 核心业务规则

3.1 版本与场景必须分层

场景是经营事实容器，版本是追溯与发布维度，不能混成同一概念。

3.2 发布必须有状态

关键对象一旦进入正式口径，应至少具有草稿、审核、激活、归档等状态中的一部分。

3.3 激活版本应可唯一识别

每个激活版本至少应具备唯一标识，用于后续追溯、比较和审计。

3.4 审计信息不能后补想象

谁创建、谁发布、何时发布、为何关闭等信息应作为对象字段或事件记录保留，而不是靠口头说明。

4. 核心对象规格

4.1 版本对象

至少应包含：

  • versionId
  • projectId
  • scenarioId（如适用）
  • objectType
  • objectRef
  • status
  • traceId
  • createdAt
  • createdBy
  • activatedAt
  • archivedAt
  • remark

4.2 发布事件对象

建议独立维护，至少包含：

  • publishEventId
  • versionId
  • actionType
  • fromStatus
  • toStatus
  • operator
  • operatedAt
  • reason

5. 状态设计建议

首期建议至少预留：

  • 草稿
  • 预检查
  • 审核中
  • 待签发
  • 生效中 / 已激活
  • 已归档
  • 已关闭
说明：首期不一定全部页面化实现，但状态结构建议先留位。

6. 与其他模块的关系

  • 与 Settings：费率、基准参数需要版本化
  • 与场景：场景可引用版本，但不替代版本
  • 与 BOM：BOM 变化可产生版本差异
  • 与报价：正式报价结果需要版本追溯基础
  • 与跟踪 / 预警：异常关闭、状态切换可引用治理状态

对吉利客户，版本治理首要覆盖内部核算、场景、参数快照和内部报价结果等系统主对象；客户最终报价单如需被引用，应仅作为对照或业务参考信息，不应作为系统正式计算内核的主版本对象。

7. 功能范围

7.1 首期必须具备的能力

  • 版本对象存在
  • 关键状态存在
  • Trace ID 或等价唯一标识预留
  • 能标记当前有效版本

7.2 第二阶段建议补齐的能力

  • 发布流页面化
  • 状态切换记录
  • 审计查看
  • 版本比较
  • 异常关闭机制联动

8. 待确认项

  • 1. 发布流节点是否完整采用历史五节点
  • 2. 首期哪些对象必须纳入版本治理
  • 3. 是否首期引入审批角色
  • 4. 审计查看是否需要独立页面

9. 开发实施建议

  • 1. 先统一版本对象结构
  • 2. 再明确场景、BOM、费率、报价等对象的版本挂接方式
  • 3. 再实现状态流与 Trace ID
  • 4. 最后补页面化治理能力

10. 验收建议

A 级验收

  • 有版本对象
  • 能区分状态
  • 能标记当前有效版本

B 级验收

  • 有状态切换记录
  • 能进行基本追溯
  • 能支撑关键对象版本引用

C 级验收

  • 审批流
  • 独立审计页
  • 复杂版本比较

11. 结论

版本与发布治理决定了系统能不能从“页面算出来”升级为“正式可管理、可审计、可追溯”的经营平台，是后续治理能力建设的底座之一。

功能规格-系统设置与参数治理

适用范围：高压线束精算与决策引擎
来源依据：PRD_统一版_高压线束精算与决策引擎.md、PRD_统一版_研发实施清单.md
文档目标：把“系统设置与参数治理”从 PRD 级表述下钻为可供研发拆解的专项功能规格
当前判断：设置页入口已存在，但参数治理与发布闭环仍属于【已确定未完全实现】

交叉引用

  • 对应总纲章节：11. 系统设置与参数治理
  • 对应开发任务书：04-开发任务书/T11-系统设置与参数治理.md

1. 功能定位

系统设置不是参数堆放页，而是系统业务口径、默认规则、基准参数与治理能力的配置入口。

它的核心价值不在于“能配很多东西”，而在于：

  • 让关键参数逐步配置化
  • 让参数有归属、有边界、有版本
  • 避免规则长期散落在硬编码中

2. 业务目标

  • 管理人工、制造、管理、利润等关键费率
  • 管理金属价格、预警阈值、默认模板等关键参数
  • 支撑多工厂、费用分摊、BOM 分类规则等配置能力
  • 为版本化、发布流、权限边界提供入口基础

3. 核心业务规则

3.1 参数必须区分层级

至少应区分：

  • 全局基准参数
  • 项目级参数
  • 场景级快照参数

3.2 参数不能只停留在实时值

对影响经营结果的重要参数，应支持快照、版本或引用关系，而不能完全依赖当前实时值。

3.3 Settings 不应变成泛化配置堆场

设置页应服务于明确业务对象与规则，不应演变成脱离主业务事实的通用配置仓。

3.4 关键参数应可治理

关键参数至少要考虑：

  • 谁能改
  • 谁能发布
  • 变更是否可追溯

4. 参数范围

当前至少覆盖：

  • 人工费率
  • 制造费率
  • 废品率
  • 管理费率
  • 利润率
  • 默认年降率
  • 铜价
  • 铝价
  • 预警阈值
  • 默认模板类型
  • 多工厂相关参数
  • 费用分摊相关参数
  • BOM 分类规则

5. 核心对象规格

5.1 参数项对象

至少应包含：

  • configItemId
  • configType
  • name
  • value
  • scope
  • status
  • versionRef
  • updatedAt
  • updatedBy

5.2 参数快照对象

建议预留：

  • configSnapshotId
  • projectId
  • scenarioId
  • snapshotType
  • snapshotPayload
  • createdAt

6. 页面与交互要求

6.1 设置主页面

至少支持现有多 Tab 结构继续承载：

  • 基础配置
  • 成本结构
  • 多工厂管理
  • 费用分摊
  • 系数近似
  • BOM 分类规则

6.2 参数展示要求

至少支持：

  • 参数名称
  • 当前值
  • 参数作用范围
  • 最近修改信息
  • 状态或版本预留

6.3 参数治理预留

后续建议支持：

  • 发布状态
  • 锁定状态
  • 变更记录
  • 生效范围说明

7. 数据与计算要求

7.1 参数引用要求

成本、报价、预警、分摊、场景等模块引用参数时，应有稳定来源，避免各自维护一份不同逻辑。

对于吉利客户，Settings 负责维护内部核算所需的关键参数稳定来源及快照能力；若客户最终报价单中出现相关费用口径，系统仅在必要时做参考对照，不要求按客户最终模板字段建立一一映射关系。

7.2 快照要求

对影响场景、报价、分摊回收等结果的关键参数，应支持快照或等价引用关系。

7.3 配置化边界

适合配置化的应进入 Settings；属于具体业务对象事实的，不应错误塞进全局配置。

8. 模块边界

  • Settings 负责参数入口与治理基础，不替代具体业务模块本身。
  • Settings 不负责直接承载项目事实、场景事实、设变事件等对象。
  • Settings 可提供参数来源，但最终结果仍应由对应模块计算和呈现。

9. 待确认项

  • 1. 哪些参数必须纳入发布流
  • 2. 哪些参数允许项目级覆盖
  • 3. 哪些参数允许场景级快照覆盖
  • 4. 财务发布权是否首期产品化
  • 5. 参数变更是否需要审批

10. 开发实施建议

  • 1. 先梳理现有设置项分类
  • 2. 再区分全局 / 项目 / 场景三层作用域
  • 3. 再补参数快照与版本引用关系
  • 4. 最后补权限、发布与追溯能力

11. 验收建议

A 级验收

  • 现有设置项可稳定查看与维护
  • 参数有明确分类
  • 参数能被主链路模块引用

B 级验收

  • 参数有作用域区分
  • 关键参数能形成快照或版本引用
  • 能支撑后续治理能力扩展

C 级验收

  • 审批发布
  • 复杂权限治理
  • 精细化变更审计

12. 结论

系统设置与参数治理的价值，不在于配置项数量，而在于把关键经营规则逐步从硬编码、散落逻辑和不可追溯状态中抽离出来，形成可维护、可版本化、可治理的统一参数基础。

1. 功能定位

Profile / 个人中心负责承载用户个人维度的信息与偏好能力，用于补齐系统从“业务模块集合”走向“完整产品”的个体使用侧能力。

它不属于经营闭环主链路，但属于产品化收口的重要一环。

2. 业务目标

  • 承载个人信息
  • 承载主题偏好或界面偏好
  • 预留权限概览、通知概览、使用配置等能力
  • 作为个人级设置入口，而不是业务规则入口

3. 核心业务规则

3.1 Profile 负责个人维度，不替代系统设置

系统参数、业务规则、全局配置仍属于 Settings，不应被放入个人中心。

3.2 Profile 可承载偏好，但不应承载重业务闭环

个人中心应避免演变成另一个混杂业务治理页面。

3.3 与权限相关内容应以“概览”优先

如后续涉及权限、通知、审计入口，首期更适合先做概览与入口，而非直接承载复杂治理操作。

4. 核心对象规格

建议至少包含个人配置对象：

  • profileId
  • userId
  • displayName
  • email
  • themePreference
  • uiPreference
  • notificationPreference
  • updatedAt

5. 页面与交互要求

5.1 基础信息区

至少支持：

  • 用户名称
  • 邮箱或账号信息
  • 基础身份展示

5.2 偏好区

至少支持：

  • 主题偏好
  • 界面展示偏好
  • 使用配置预留

5.3 概览区预留

后续建议预留：

  • 权限概览
  • 通知概览
  • 会话或使用状态概览

6. 数据与边界要求

  • Profile 数据应与系统全局配置分离
  • 个人偏好不应覆盖业务口径参数
  • Profile 可提供主题或界面偏好来源，但不应影响正式经营计算

7. 模块边界

  • Profile 属于个人使用层，不替代 Settings。
  • Profile 不承载场景、报价、分摊、跟踪等主业务对象。
  • Profile 可作为产品化补充入口，但不是主经营链路的优先模块。

8. 待确认项

  • 1. 首期只做个人偏好，还是同时纳入权限概览
  • 2. 是否纳入通知中心入口
  • 3. 是否纳入审计或会话入口
  • 4. 是否与主题改造同步推进

9. 开发实施建议

  • 1. 先定义个人配置对象
  • 2. 先落个人信息与偏好基础能力
  • 3. 再决定是否补权限 / 通知 / 审计入口概览

10. 验收建议

  • 有页面入口

  • 能保存个人偏好

B 级验收

  • 能与系统整体主题能力联动

C 级验收

  • 权限概览
  • 通知概览
  • 更重的个人工作台能力

11. 结论

Profile / 个人中心的价值，不在于增加一个附属页面，而在于补齐系统从“业务功能集合”到“完整产品体验”的最后一层个人使用侧能力。但它应始终服从主经营闭环优先级，不抢核心业务主线。

但这份报价单是销售最终按商务策略调整后、提交到客户系统并被客户承认的结果文件。它不应被系统内核直接建模为正式可计算对象，更适合作为内部核算结果的对照参照物，重点参考其中的分摊费用及回收方式、出厂价、到厂价和定点确认口径。

5. 真实字段语义

5.1 价格汇总页是客户最终报价主表

价格汇总 的核心口径非常明确：

  • 总到厂价（未税） = 客户最终看到的总价口径
  • 出厂价（不含分摊） = 直接材料 + 制造费用 + 期间费用及利润
  • 1.直接材料 下又拆为：
  • 原材料成本（A1）
  • 外购件成本（A2）
  • 2.制造费用 下至少包括：
  • 加工费用（B1）
  • 废品损失（B2）
  • 3.期间费用+利润 下至少包括：
  • 管理费用（C1）
  • 财务费用（C2）
  • 销售费用（C3）
  • 利润（D）
  • 4.总投资费用 当前模板首列展示为总工装费用

这说明吉利模板的真实报价核心不是此前写的“铜重、铝重、工时、包装费、运输费”那套结构，而是按客户模板要求分成直接材料、制造费用、期间费用+利润、总投资费用四大块。

5.2 工装费不是一个简单汇总数

4.1工装费 不是只给总额，而是展开到：

  • 工装分类
  • 子零部件编号/名称
  • 新开/借用
  • 工序内容
  • 工序号
  • 工装名称
  • 模穴数
  • 寿命
  • 外形尺寸
  • 设计费用
  • 上模/下模品牌与费用

所以系统里不能只保留 totalToolingFee 一个数字，至少要保留工装明细层。

5.3 试验费按项目逐项展开

4.2.试验费 的总试验费用公式为：

总试验费用 F = (每次试验费 A + 样件数量 C × 样件价格 D + 试验工装费用 E) × 试验次数 B

这说明试验费对象本身就是结构化计算对象，不是备注字段。

5.4 研发费是多费用篮子相加

4.3.研发费 的总研发费用公式为：

总研发费用 P = A + B + C + D + E + F

其中：

  • A = 硬件开发费用
  • B = 软件开发费用
  • C = 标定测试费用
  • D = 开发工具费用
  • E = 认证费用
  • F = 差旅费

所以研发费不是单列金额，而是由多种研发活动汇总而成。

6. 真实计算链路

6.1 价格汇总主链路

模板中已经直接给出主公式定义：

出厂价（不含分摊） P1 = A1 + A2 + B1 + B2 + C1 + C2 + C3 + D
总到厂价（未税） P2 = P1 + E1 + E2 + F1 + F2 + G1 + G2 + J1 + J2 + K + L

其中当前可直接确认的展示层为：

  • A1 原材料成本
  • A2 外购件成本
  • B1 加工费用
  • B2 废品损失
  • C1 管理费用
  • C2 财务费用
  • C3 销售费用
  • D 利润
  • 4.总投资费用中至少有工装费

说明模板本身保留了更完整的客户报价分项，只是当前截取区未完全展开 E/F/G/J/K/L 各项中文标题。

6.2 工装费链路

工装费明细
→ 按工装分类/工序/子零部件录入
→ 形成线束号级工装费用
→ 汇入价格汇总中的总投资费用

6.3 试验费链路

试验项目明细
→ 计算单项总试验费用
→ 按零件号汇总
→ 进入总到厂价相关投资/附加费用口径

6.4 研发费链路

研发工作内容明细
→ 分别计算硬件/软件/标定/工具/认证/差旅
→ 汇总总研发费用
→ 进入报价总成本口径

7. 吉利报价单与内部核算表的边界

7.1 内部报价核算表才是系统主对象

像：

  • 吉利E281报价核算_BOM核对版.xlsx
  • 吉利E281定点核算_BOM核对版.xlsx

这类文件对应的才是系统真正要承载和管理的内部核算逻辑。

7.2 吉利最终报价单不是系统可完全映射对象

吉利最终报价单是销售按商务策略调整后的最终提交版本，其中很多费用表达和利润安排已经过商务处理，和内部核算不存在稳定一一映射关系。

因此系统不应尝试把整份吉利最终报价单完整反向建模为可计算内核对象。

7.3 系统只参考少数关键对照口径

对于吉利最终报价单，系统主要参考：

  • 分摊费用及回收方式
  • 出厂价
  • 到厂价
  • 定点确认含义

除此之外的商务调整项，不应强行纳入系统映射规则。

8. 推荐落地约束

8.1 系统以内部核算为主

DS02 用于帮助理解吉利最终报价单，不改变系统以内部核算为主对象的边界。

8.2 只保留关键对照口径

系统如需引用吉利最终报价单，应优先只保留：

  • 出厂价
  • 到厂价
  • 分摊费用表达
  • 回收方式表达
  • 是否定点/已被客户承认

8.3 不要尝试反推销售商务策略

销售最终报价中的商务调整项不应被强行拆回系统规则，也不应要求与内部核算逐字段映射。

8.4 工装分类与工序号映射仍可作为辅助参考

它属于模板辅助规则，可用于理解费用结构，但不是系统主报价结果对象。

9. 当前应补充到系统说明书中的结论

  • 1. 吉利最终客户报价单真实载体是 高压线束包1-总报价模板(新)-20251117132314103 - 定点.xls。
  • 2. 这份报价单是销售最终按商务策略调整后提交给客户并被承认的结果，不是系统主计算对象。
  • 3. 系统主对象仍然是内部核算表；吉利最终报价单主要用于参考分摊费用及回收方式、出厂价、到厂价和定点含义。
  • 4. 吉利用户口径下的很多费用调整与商务策略，不能要求与内部核算逐字段稳定映射。
  • 5. DS02 的作用应是划清“可参考口径”和“不可强映射口径”，而不是要求系统完整承载最终客户报价单。

10. 结论

吉利报价单应被视为销售最终提交给客户并被客户承认的结果文件。对系统而言，它的价值不在于被完整建模，而在于帮助确认客户最终认可的分摊费用及回收方式、出厂价、到厂价和定点口径。系统主对象仍应坚持内部核算结构，不应把销售最终报价策略反向硬映射进内核。

2. 真实 BOM 工作簿的结构定位

从 吉利E281 国内项目 TT BOM V05-2026.01.04.xlsx 与 E281项目 报价BOM V01-11.3.xlsx 看，真实 BOM 工作簿至少包含以下几层：

  • 1. 配置清单：线束号与车型/配置/适配关系

  • 4. 二次物料明细：从 KSK 明细中反查并汇总出的二次物料层
  • 5. KSK线束BOM明细：项目级、可汇总的线束 BOM 主表
  • 6. 单线束 sheet（如 6608516992）：某条线束的原始结构展开页

因此，真实 BOM 不是单个明细表，而是一个从配置层到单线束展开层的多 sheet 结构体系。

来源于 二次物料明细，是对 KSK 明细中的某些物料描述进行汇总后的结果层。

5.1 配置清单不是 BOM 明细本身

  • 哪些线束存在
  • 对应什么零件号/名称
  • 是标配还是选配
  • 适用于哪些车型配置

它是线束选择层，不是材料明细层。

5.2 KSK线束BOM明细才是项目级 BOM 主表

从真实引用关系看：

  • 报价核算表 配置明细 用 SUMIFS(KSK线束BOM明细!...)
  • 二次物料明细用 VLOOKUP/SUMIFS(KSK线束BOM明细!...)

说明 KSK线束BOM明细 才是最关键的 BOM 事实来源。

5.3 单线束 sheet 是原始展开页，不等于系统主存储结构

如 6608516992 sheet 中还保留：

  • Function
  • Wire NO.
  • PIN
  • OPTION
  • SPEC
  • Sub-Part Number/Name/Quantity

这些字段非常适合做原始解析与追溯，但在系统内核里不一定全部作为统一主对象字段暴露。

5.4 二次物料明细是派生层

其公式大量使用：

  • VLOOKUP(..., KSK线束BOM明细!D:N, ...)
  • SUMIFS(KSK线束BOM明细!K:K, ...)

说明它是从 KSK 明细中提取/聚合出的二次材料视图，而不是独立录入主表。

6. 真实计算链路

6.1 配置清单确定线束集合

配置清单
→ 给出线束号、线束名称、标配/选配、车型适用关系
→ 形成当前项目/配置下应进入核算的线束集合

6.2 单线束展开页进入 KSK线束BOM明细

真实工作簿里，单线束页保留最原始结构，而系统做项目级汇总时应优先落在 KSK线束BOM明细 这一层。

单线束原始结构页
→ 统一归并为 KSK线束BOM明细
→ 形成项目级可汇总 BOM 主表

6.3 KSK线束BOM明细向报价核算表汇总

在 吉利E281报价核算_BOM核对版.xlsx / 配置明细 中可见：

材料成本 = SUMIFS(KSK线束BOM明细!S:S, KSK线束BOM明细!A:A, 线束号)
铜重 = SUMIFS(KSK线束BOM明细!U:U, KSK线束BOM明细!A:A, 线束号)
铝重 = SUMIFS(KSK线束BOM明细!T:T, KSK线束BOM明细!A:A, 线束号)
导线成本 = SUMIFS(KSK线束BOM明细!V:V, KSK线束BOM明细!A:A, 线束号)

这说明 KSK 主表中不仅有基础物料字段，还已经包含：

  • 材料成本列
  • 铝重列
  • 铜重列
  • 导线成本列

系统设计时，不能只看到前面可见的 A-N 列，就把 BOM 简化成“料号 + 数量 + 单位”。

6.4 二次物料明细从 KSK 主表派生

二次物料描述
→ VLOOKUP 取 KSK线束BOM明细中的名称/单位/供应商/SAP
→ SUMIFS 汇总 KSK线束BOM明细中的数量

这表明二次物料更适合作为派生分析对象，而不是 BOM 主事实对象。

6.5 变更履历驱动 BOM 差异解释

变更履历 中明确记录：

  • 导线规格变化（如 50mm² → 35mm²）
  • 用量变化
  • 套管型号变化
  • 新增橡胶件、支架数据变化等

因此设变分析不能只比较两版 BOM 行差异，还要保留业务解释文本。

7. BOM 与系统模块的真实关系

7.1 与客户报价工作台

客户报价表中的材料成本、铜重、铝重、导线成本都直接来自 BOM 汇总结果。

7.2 与设变与跟踪

变更履历 和 KSK 明细共同构成设变识别基础：

  • 一部分是结构化字段差异
  • 一部分是人工归纳后的业务变更说明

7.3 与场景管理

不同 BOM 文件版本（报价 BOM、TT BOM、定点 BOM）本身就是不同场景/阶段的事实来源。

7.4 与数据结构建模

系统不能只建一个 BomRow，至少要区分：

  • 配置层
  • KSK 主明细层
  • 单线束原始层
  • 派生汇总层
  • 变更层

8. 推荐实现约束

8.1 以 KSK线束BOM明细作为系统主 BOM 来源

因为真实报价核算表就是按这一层做汇总。

8.2 单线束页保留为可追溯原始结构

不要直接丢弃，但也不要把它直接等价成系统统一主表。

8.3 配置清单必须独立建模

标配/选配、车型适用关系、线束集合选择都在这里，不应混进 BOM 明细行。

8.4 变更履历需要同时保存结构差异与业务描述

否则无法还原真实业务对 BOM 变化的解释口径。

8.5 BOM 汇总层必须保留材料成本/铜重/铝重/导线成本等结果字段

因为下游报价核算直接按这些字段聚合，而不是再逐个原始散件重算。

9. 当前应补充到系统说明书中的结论

  • 1. 真实 BOM 工作簿是“配置清单 + KSK主明细 + 单线束页 + 二次物料 + 变更履历”的组合结构。
  • 2. KSK线束BOM明细 是当前最关键的项目级 BOM 主事实来源。
  • 3. 单线束页适合做原始追溯，不应直接替代项目级 BOM 主表。
  • 4. 报价核算表已经依赖 BOM 汇总结果字段（材料成本、铜重、铝重、导线成本）而不是只依赖原始散件字段。
  • 5. BOM 模型必须同时支持配置选择、结构追溯、汇总核算和设变解释。

10. 结论

基于真实 E281 BOM 工作簿，BOM 表不能再被抽象成一个简单的 BomRow 列表，而应被理解为从配置选择、单线束展开、KSK 主明细、二次物料派生到变更履历解释的多层结构体系。系统只有按这个真实层次建模，后续报价、设变、跟踪和版本治理才能接上真实业务。
