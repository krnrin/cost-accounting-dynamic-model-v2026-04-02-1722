## 一、产品定位与价值主张

### 1.1 产品愿景

**一句话定位**: 面向汽车高压线束 Tier1 供应商的「单线束号级精准报价 + 跨部门协作审批」双模应用 — 离线单机即开即用，联网自动升级为飞书协作平台。

**核心价值**:
- **报价精度**: 从项目级系数近似 → 单线束号 BOM 逐行精算，误差从 ±15% 降至 ±2%
- **协作效率**: 从 Excel 邮件来回 → 飞书内「录入→核算→审批→发布」全流程，报价周期从 5天 → 1天
- **变更响应**: 设变/金属联动自动计算差异，生成客户报价单，从 2天 → 2小时
- **双模部署**: 无网络时完整离线运行（本地存储+本地核算），有网络时自动同步到云端+飞书协作

### 1.2 目标用户

汽车高压线束 Tier1 供应商内部跨部门团队，服务于 OEM 定点报价和设变报价场景。

### 1.3 与行业现有工具的差异

| 维度 | Excel 手工 | ERP 成本模块 | 本产品 |
|:---|:---|:---|:---|
| 核算颗粒度 | 单线束号（但手工维护） | 通常到总成级 | 单线束号（自动化） |
| 变更响应速度 | 半天~1天 | 需跨模块操作 | 实时联动 |
| 协作模式 | 邮件传 Excel | 系统内审批 | 离线单人/在线飞书协作 |
| 离线能力 | ✅ (Excel本身) | ❌ (必须联网) | ✅ (PWA离线完整可用) |
| 客户报价单 | 手工填吉利/比亚迪模板 | 通常不支持 | 自动生成客户模板 |
| 金属联动 | 手工重算 | 需定制开发 | 内置联动引擎 |
| 学习成本 | 低（Excel） | 高（需培训） | 低（类Excel交互，Univer引擎） |

## 二、行业背景与协作流程分析

### 2.1 高压线束行业特点

高压线束是新能源汽车的「血管系统」，连接电池包、电驱、OBC、PTC、压缩机等核心高压部件。相比传统低压线束，高压线束有以下**独特的成本特征**：

| 特征 | 对报价工具的影响 |
|:---|:---|
| **铜/铝用量大** — 单根高压线缆截面积可达 70mm²，铜重占材料成本 40-60% | 必须支持金属价格联动，铜价每波动 1000 元/吨，单车成本变化 2-5 元 |
| **连接器价值高** — 高压连接器单价 30-200 元/个，占材料成本 20-35% | 连接器价格不能用系数近似，必须逐件核算 |
| **BOM 层级深** — 一个充电插座线束总成可能包含 30-50 种物料 | BOM 必须逐行解析，支持分类（导线/连接器/端子/辅料） |
| **装车比差异大** — 同一总成族有 3-5 个衍生型号，装车比从 3% 到 52.5% | 必须支持加权汇总，装车比是单车成本的核心乘数 |
| **设变频繁** — 一个项目生命周期 6 年，可能发生 20-50 次设变 | 变更报价引擎是核心功能，不是附加功能 |
| **客户模板各异** — 吉利用 A1/A2/B1/B2/C/D 结构，比亚迪/蔚来有不同格式 | 报价输出必须适配多客户模板 |
| **金属价格合同** — 通常有铜价/铝价联动条款，超出基准 ±X% 触发调价 | 需要内置金属联动计算和预警 |
| **工时分工序** — 开线/公共制程/总装三段工时，不同工序费率不同 | 工时录入需支持工序拆分 |

### 2.4 当前痛点分析

| # | 痛点 | 影响 | 本产品解决方式 |
|:---|:---|:---|:---|
| 1 | Excel 多版本失控 | 不知道哪个是最新版，数据被覆盖 | 版本控制 + 变更日志 |

| 3 | 金属联动手工重算 | 耗时且易出错 | 自动联动引擎 |

| 7 | 不同客户模板手工适配 | 每个 OEM 的报价格式不同 | 模板引擎，一键导出 |

### 5.3 模块 ②：数据录入 — 核心交互设计

**设计原则**: 线束工程师录入一个线束号的数据后，点击「发布」，系统自动通知下一个环节的负责人接力。

#### 5.3.1 BOM 录入（线束开发工程师）

**录入方式优先级**:
1. **Excel 文件上传**（推荐）— 直接上传 KSK BOM 明细 Excel，系统自动解析
2. **飞书表格粘贴** — 从飞书文档/Excel 复制表格区域粘贴
3. **手工逐行录入** — 适合少量修改
4. **从历史版本克隆** — 设变时只改变化部分

## 六、数据架构设计

### 6.1 离线优先(Offline-First)双模架构

```
                    ┌─────────────────────────────────────────┐
                    │            应用层 (同一套代码)            │
                    │                                         │
                    │   React + Univer + 核算引擎 + UI        │
                    │              │                          │
                    │         ┌────┴────┐                     │
                    │         │ 数据层   │ ← 统一 API 接口      │
                    │         │ (抽象)   │                     │
                    │         └────┬────┘                     │
                    │         ┌────┴────────────┐             │
                    │         │                 │             │
                    │    离线模式            在线模式           │
                    │    IndexedDB         IndexedDB          │
                    │    + 本地文件           ↕ 同步            │
                    │    (完整功能)       后端 API              │
                    │                       ↕                 │
                    │                   PostgreSQL            │
                    │                       ↕                 │
                    │                 飞书 API (审批/通知)      │
                    └─────────────────────────────────────────┘

关键原则:
  ① 本地永远是第一写入点 (IndexedDB)
  ② 核算引擎永远在前端运行 (不依赖后端)
  ③ 后端仅做: 同步中转 + 权限控制 + 飞书对接
  ④ 离线→在线切换: 自动增量同步，无需用户操作
```

### 6.2 核心数据模型（离线/在线共用）

前端 IndexedDB 和后端 PostgreSQL 使用**同构的数据模型**，每个文档/记录含同步元数据：

```javascript
// 每条记录的同步元数据
{
  _id: "uuid-v4",           // 前端生成，全局唯一
  _rev: "3-a1b2c3d4",      // 修订版本号 (用于冲突检测)
  _updatedAt: 1712150400,  // 最后修改时间戳
  _syncedAt: null,         // 上次同步时间 (null=未同步)
  _deleted: false,         // 软删除标记
  _createdBy: "user-id",   // 创建者 (离线时为本地ID)
}
```

├────────────────────────────────────────────────────────────┤
│                                                            │

│  │  project_code, project_name, customer_id, platform      │

### 6.3 同步策略

```
┌──────────────────────────────────────────────────────────────┐
│                    增量同步协议                                │
│                                                              │
│  上行 (前端 → 后端):                                         │
│    1. 前端收集 _syncedAt < _updatedAt 的记录                 │
│    2. 按 _updatedAt 排序，批量 POST /sync/push              │
│    3. 后端检查 _rev，无冲突则写入 PG，返回新 _rev            │
│    4. 前端更新 _syncedAt                                     │
│                                                              │
│  下行 (后端 → 前端):                                         │
│    1. 前端发送 GET /sync/pull?since={lastSyncTimestamp}      │
│    2. 后端返回该时间点后其他用户的变更                         │
│    3. 前端合并到 IndexedDB                                    │
│                                                              │
│  冲突策略 (发生概率极低):                                     │
│    · 同一线束号同一字段：后写入者提示，用户选择保留哪个         │
│    · 不同线束号：自动合并，无冲突                              │
│    · BOM行级：行ID不同则自动合并，相同行ID则提示               │
│                                                              │
│  触发时机:                                                    │
│    · 应用启动时（如果在线）                                    │
│    · 每 30 秒静默轮询                                         │
│    · 用户点击「同步」按钮                                      │
│    · 网络恢复时（navigator.onLine事件）                        │
└──────────────────────────────────────────────────────────────┘
```

### 9.4 核算引擎策略（前端为主）

```
关键决策：核算引擎始终在前端运行

原因:
  ① 离线单机版必须能完整核算 → 引擎必须在前端
  ② 避免前后端双份引擎的维护负担
  ③ 数据量小（11个线束 × 50行BOM），前端性能绰绰有余

  engine/ (纯JS，无DOM依赖，前后端共用)
  ├── harness_costing.js   ← 新增：单线束号级核算
  ├── change_pricing.js    ← 新增：变更报价引擎
  ├── metal_escalation.js  ← 新增：金属联动
  ├── compute_model.js     ← 重构：年度P&L (调用harness_costing)
  ├── bom_parser.js        ← 保留：BOM解析
  └── quote_template.js    ← 新增：客户报价模板映射

在线模式下，后端也可 require() 同一份引擎做校验性核算:
  后端核算 = 前端核算的校验副本，不是权威源
```

### 9.5 离线分发方式

```
方式 A: PWA (推荐)
  · 用户首次在线访问后，Service Worker 缓存全部资源
  · 之后断网也能打开和使用
  · 恢复网络自动同步
  · 无需安装

方式 B: Electron 打包 (可选)
  · 打包为桌面应用 (.exe / .dmg)
  · 适合完全无网络的工厂场景
  · 同一套 React 代码，仅加 Electron Shell
  · 数据存本地 IndexedDB (或 SQLite via better-sqlite3)

方式 C: 静态文件部署 (兜底)
  · vite build → dist/ 文件夹
  · 复制到本地，双击 index.html 即可运行
  · 最简单，但无自动更新能力
```

## 附录 B：客户报价模板适配

### 吉利模板结构

```
A1 原材料 + A2 外购件
B1 加工费(制造费) + B2 废品损失
C1 管理费(4%) + C2 财务费(4%) + C3 销售费(4%)
D 利润(4%)
E 工装(分摊) + F 试验(分摊) + G 研发(分摊)
```

### 映射关系

```
A1 = Σ(BOM中导线类行的材料成本)
A2 = Σ(BOM中连接器/端子/辅料的材料成本)
B1 = 实际工时 × 制造费率
B2 = (A1+A2) × 废品率
C1 = (A1+A2+B1+B2) × 4%
C2 = (A1+A2+B1+B2) × 4%
C3 = (A1+A2+B1+B2) × 4%
D  = (A1+A2+B1+B2) × 4%
E  = 模具投入 / 分摊量
F  = 试验投入 / 分摊量
G  = 研发投入 / 分摊量
```
