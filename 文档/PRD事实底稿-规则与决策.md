# PRD事实底稿-规则与决策

## 已确认正式规则
### 产品定位
- 产品定位已从“报价工具”演进为“高压线束精算与决策引擎”。
- 系统核心价值不止于生成客户模板，而是统一承载内部实绩成本管理与客户报价决策。

### 核算颗粒度
- 核算颗粒度以单线束号 / BOM 行为主，而非项目级、套级笼统核算。
- PRD 与逻辑说明书都反复强调逐行 BOM、逐条线束、逐层汇总。

### 双引擎
- 系统采用“内部实绩成本 + 客户报价”的双线核算口径。
- 同一条线束需要同时看到内部真实成本与客户报价结果，用于利润穿透与调价判断。

### 一次性费用分摊
- 一次性费用（工装/试验/研发）必须按线束号独立、按根分摊。
- 这是相对旧版“按套笼统分摊”的关键升级点。

### 分摊回收跟踪
- 系统需要基于装车比 × 累计产量跟踪分摊回收进度。
- 分摊完毕后要能支持调价提醒。

### 多场景管理
- v2.1 主 PRD 已明确一个项目多场景模型：初始报价、定点、设变、年降。
- 各场景独立持有生命周期、产量、费率、BOM、分摊状态等关键业务事实。

### 金属价格联动
- 铜/铝价格是成本敏感因素，系统必须支持金属价格联动。
- 逻辑说明书中已明确金属价格来源与公式位置，说明这不是概念性需求，而是已有业务依据的规则。

### 决策支持定位
- 产品必须支持比较、追踪、预警、回溯，而不是单纯填表或导出工具。

## 程序状态相关决策
- “程序完成状态”必须作为新 PRD 的主依据之一，而不是仅做边缘参考。
- 但程序现状要以当前主应用 `D:/成本核算动态模型/app` 的真实代码为准。
- 历史会话或其他分支中出现、但当前主应用未确认的页面/能力，不能直接写成“已实现”。

## 历史会话已拍板结论
- 正文只保留稳定结论，不把历史讨论原样拼接进主 PRD。
- 统一新 PRD 必须同时兼容：
  - 产品总纲
  - 可开发规格
  - 对外汇报表达
- `cloud-code` 仓库不是成本核算系统本体，不能混淆。
- 成本核算项目主目录应看 `D:/成本核算动态模型`。
- 历史上下文的重要来源包括：
  - `D:/成本核算动态模型/文档`
  - `C:/Users/lyvee/.accio/accounts`

## 当前争议项与待确认项
- 场景之间的数据继承机制
- 报价引擎中哪些参数由销售可调、哪些由财务锁定
- 预警系统分级标准（阈值、颜色、触发逻辑）
- 个人中心 / Profile 的职责边界
- 设变与年降的版本管理深度
- 客户模板适配范围与优先级

## 历史演进说明
- v1.x 关注“客户报价单生成”。
- 2026-04-02 的历史会话上下文仍明确“工厂端离线报价协作工具、报价版优先、单线束颗粒度、一次性费用不按总部平均分摊”，说明旧版本主视角更偏报价执行与工厂项目经理口径。
- v2.0 关注“实绩成本管理平台”，引入双引擎、按根独立分摊、分摊回收跟踪。
- v2.1 继续推进到“一个项目多场景模型”，并增加 UI 工业蓝图风、预警系统、个人中心等产品层内容。
- 因此新 PRD 不应回退到“单次报价输出工具”的旧定位，但可保留“报价执行协同”作为场景之一，而不是产品全部定位。

## 开发指南与历史补丁反映出的稳定方向
- 旧架构阶段已经明确拆分为预演 / 核算 / 跟踪 / 归档四页心智模型，这说明产品并非单页报价单，而是围绕决策、核算、执行跟踪、版本归档展开。
- 多项目复用方案中已经出现“项目注册表 + 独立 IndexedDB + 项目切换器 + 关键参数锁定/变更流程”，说明“一个项目一个独立数据域”和“变更受控”是稳定方向。
- G281 落地验收清单已把 RBAC、基准发布流、Audit Trace、异常关闭、协议价落实、多工厂对标纳入验收，说明权限、流程、审计、预警并非装饰需求，而是产品治理能力的一部分。
- 若干 issue 补丁指南持续强化几项业务纠偏：
  - 进度价本质是“协议价 vs 当前批量价”的差距追踪，不是加权混合价。
  - 残余材料池本质是“变更取消料号的呆滞提报”，不应继续分摊进当前产品成本。
  - 利润归因应保留 Shapley，同时增加因果链瀑布图，说明系统需要兼顾数学归因与业务可解释性。
  - 金属敏感度、材料组成、状态默认值应配置化，说明关键参数不应长期硬编码。
  - Schema 迁移、工具函数去重、引擎拆分、文件去 `g281_` 前缀与模块化迁移，说明系统方向已经从单项目脚本式页面转向可复用、可演进的平台化结构。
- 历史会话正文进一步确认：
  - 项目曾被概括为“单线束号级精准报价 + 跨部门协作审批”的双模应用，且支持离线单机与飞书在线协作两种模式。
  - 飞书集成方向曾明确包括：飞书免登、OAuth、Bitable 数据同步、消息卡片、审批流，这些可视为“协同化扩展方向”，但不应覆盖主产品的本地可运行定位。
  - 历史任务清单与会话快照中曾出现更高完成度基线（0 TS errors / 30 test files / 271 tests / vite build ✅），说明“历史完成状态快照”与“当前主应用现实”必须分层写入 PRD。
  - E281 数据注入与校验会话表明，产品已不再停留在 G281 单项目演示，而是明确朝多项目复用、项目数据导入与校验演进。
  - 至少两条 Claude resume 会话都出现了基于 Playwright 的真实页面巡检，说明“页面可访问性与核心流程联通”在历史上曾被作为程序状态的重要验证方式。
  - 其中 `Alerts/Profile` 在历史会话中属于明确补全目标而非天然稳定模块，因此新 PRD 对这两块应谨慎采用“已确定未完全实现”或“待确认”的表述。
- 这些内容更多体现为“业务规则与产品方向已确定”，但是否已在当前 `app/` 全量落地，仍需以主应用代码核准后分级表述。

## 当前写入新PRD的原则
- 已被文档、会话、程序三方共同支持的内容，可写为正式规则。
- 只有文档/会话支持、但程序尚未完全体现的内容，写为“已确定未完全实现”。
- 当前主应用代码中无法确认、且历史记录存在版本差异的内容，写为“待确认”。
